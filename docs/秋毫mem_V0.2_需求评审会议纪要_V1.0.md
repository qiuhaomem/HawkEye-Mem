# 秋毫mem V0.2 · 需求清单 多方评审会议纪要

**评审对象**：V0.2 需求调研报告 + 五级需求分解表
**会议主持**：高级产品经理
**参与方**：技术负责人、市场运营、法务、财务、PMO
**创始人列席**
**结论性质**：通过，含 9 条整改要求


### 👤 技术负责人

需求整体可落地。三个具体问题：

**第一，GPU 采集的 NVML 绑定方案需要提前验证。** 需求里写了"NVML直接采集或nvidia-smi解析"，但 NVML 的 Rust 绑定在 musl 静态链接环境下的兼容性是未知的。我建议在 V0.2 W1 就做一个最简可行性验证：用 NVML 绑定拿到显存总量，能在 musl target 下编译通过并运行，就继续；如果卡住超过 1 天，直接切换为解析 `nvidia-smi --query-gpu` 的 CSV 输出。**需要在需求文档里明确这个止损条件。**

**第二，`ResourceCollector` trait 的重构范围被低估了。** V0.1 的 `MemoryCollector` 只返回内存指标。V0.2 要统一返回内存+磁盘+CPU+GPU。如果每个 Collector 独立返回自己的数据结构，上层组合逻辑会非常混乱。建议在 V0.2 技术方案设计时，先定一个统一的 `ResourceSnapshot` 结构体，所有 Collector 都往里填自己负责的字段，没采集的字段留 `None`。这比需求文档里写的"重命名 trait"改动更大，但架构收益更高。

**第三，`--compare` 多模型对比是一个隐藏的性能坑。** 需求里写了对比两个模型，但如果用户对比 5 个模型、每个评估 4 项资源，采集逻辑会串行跑 20 次。建议限制 `--compare` 最多 3 个模型，且共享一次系统采集结果，不重复采。

**结论**：通过。要求明确 GPU 止损条件、技术方案时重新设计统一结构体、`--compare` 限制 3 个模型。


### 👤 市场运营

"Agent时代的鲁大师"这个概念在需求文档里一个字都没出现。需求文档是给开发看的，但里面应该有一句简单的价值陈述，让开发知道自己在为什么而写代码。建议在需求文档开篇加一句话："V0.2 的核心叙事是'Agent 时代的鲁大师——住进去之前先看房'。"

**第二个问题**：V0.2 的传播节奏没有在需求里体现。我需要在 V0.2 开发启动时同步准备三件事：
- 一张 V0.1 vs V0.2 功能对比图
- 一个 `--can-run` 的终端录屏 demo（15 秒）
- 在 V0.2 README 中嵌入一个 GitHub Issue，征集"你希望秋毫mem 下一步监控什么资源"

这些不是开发任务，但需要在版本时间表里预留我的准备窗口。

**第三个建议**：`--list-models` 输出如果只是纯文本列表，传播价值为零。建议输出一个带分类的彩色表格，包含模型名、参数量、推荐量化和最低内存要求。这个表格本身就是社区愿意截图转发的素材。

**结论**：通过。要求在需求文档中加价值陈述、发布时配套对比图和 demo、`--list-models` 输出格式设计为可传播的彩色表格。


### 👤 法务

`--can-run` 这个功能有一个新的责任敞口，之前的 DISCLAIMER 没有完全覆盖。

当前的免责声明主要针对"运行中建议"。但 `--can-run` 给出的是"可以部署"或"建议用这个配置部署"的判断。如果用户因为秋毫mem 说可以部署，下载了 50GB 模型，花了几个小时，结果跑不起来——这个损失比 OOM 更具体、更可量化。

我需要在 V0.2 发布前，在 DISCLAIMER 中增加一条明确针对部署评估的条款，大意是：
> "`--can-run` 输出的部署建议基于有限信息估算。实际部署是否成功还取决于驱动版本、系统配置、运行时负载等本软件无法预知的因素。因采纳部署建议而产生的任何损失（包括但不限于带宽消耗、下载时间浪费、存储空间占用），本软件作者不承担责任。"

另外，内置的模型参数库（8 个模型）的数据来源需要标注清楚。如果参数是从官方文档或社区实测中获取的，应该在文档中说明数据来源和最后更新时间。这不是严格的法律要求，但可以避免"你们内置的数据不准导致我部署失败"的纠纷。

**结论**：通过。要求 V0.2 发布前更新 DISCLAIMER，增加部署评估免责条款。模型参数库标注数据来源和更新时间。


### 👤 财务

需求清单里 GPU 是实验性的，磁盘和 CPU 是主线。这个优先级和我之前提的商业化验证节奏一致。

但我需要提醒一个隐性成本：GPU 采集如果要支持 NVML 直接绑定，可能会引入对 NVIDIA 专有库的链接依赖。如果这些库的许可证和我们 Apache-2.0 不兼容，就需要把 GPU 功能拆成独立插件或以动态链接方式处理。建议在技术方案阶段做一次许可证兼容性检查。

另外，模型参数库如果后续要扩展到社区共建（需求里写了 `--contribute-model`），就会产生维护成本——谁来审核社区提交的参数？标准是什么？建议在 V0.2 中先把这个入口做出来，但标注"社区贡献功能处于实验阶段，参数准确性由贡献者自行负责"，把维护成本控制在我们可承受的范围内。

**结论**：通过。要求技术方案时检查 GPU 依赖许可证兼容性。社区贡献入口标注实验性。


### 👤 PMO

需求清单的粒度足够进入技术方案设计了。但有三个工期相关的问题需要明确：

**第一，GPU 实验性的工期不确定性最高。** 技术负责人提的"W1 做可行性验证，卡住就切换方案"是正确的。但即使切换方案（解析 nvidia-smi），集成测试仍然需要真实 NVIDIA 环境。如果 CI 中没有 GPU runner，手动测试时间需要提前锁定。

**第二，`ResourceCollector` trait 重构涉及 V0.1 的所有代码。** 这不是一个轻量改动。建议在 W1 的前两天专门做这件事，重构完成后立刻跑全量测试，确认没有破坏 V0.1 的内存监控功能。不要和其他功能并行开发。

**第三，需求中"多 Agent 检测"被标记为 P2（Could Have）。** 我建议直接砍掉，放到 V0.3。因为 V0.2 已经新增了 3 个 Collector + `--can-run` 评估引擎 + 模型参数库，功能已经足够多了。多 Agent 检测放在 V0.2 只会增加测试复杂度，不如集中精力把磁盘和 CPU 做稳。

**结论**：通过。要求 W1 前 2 天专门做 trait 重构并回归测试。建议砍掉多 Agent 检测，移到 V0.3。
