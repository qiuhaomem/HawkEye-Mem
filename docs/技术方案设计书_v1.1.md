# 秋毫mem · 技术方案设计书 V1.1（修订版）

**修订记录**：
- V1.0：初始版本，基于 Cursor + Claude Code 的独立编码工具链。
- **V1.1**：采纳创始人决策，引入 Hermes 总调度 + DeepSeek-TUI 编码执行，形成多 Agent 协作模式。

---

## 一、AI 编程工具选型（更新）

本次选型以 **Hermes 为核心调度器**，**DeepSeek-TUI 为主要编码执行 Agent**，形成"大脑 + 手"的分工模式。

| 角色 | 工具 | 用途 | 选型理由 | 预算 |
|------|------|------|----------|------|
| **总调度 Agent** | **Hermes** | 任务拆解、上下文注入、跨 Agent 协调 | 作为项目框架，Hermes 可同时调度秋毫mem（内存感知）和编码 Agent，形成完整决策-执行闭环 | 已部署 |
| **主编码 Agent（核心逻辑）** | **DeepSeek-TUI** | Rust 核心模块编码（trait、平台封装、状态机、估算引擎） | 原生支持 Rust，MCP 协议集成；RLM 模式可并行调用子 Agent；模型成本极低 | ¥30-50/月 |
| **主编码 Agent（样板/文档/测试）** | **DeepSeek-Reasonix** | 脚手架、配置解析、单元测试、README、CLI 参数 | 缓存命中率高，成本更低；适合样板代码和结构化任务 | ¥10-20/月 |
| **辅助/审查工具** | **Cursor Pro** | 局部代码补全、手动编辑、最终审查 | 用于人工干预和关键逻辑的深度审查，确保 AI 生成代码的架构一致性 | $20/月 |
| **依赖审查** | **cargo-bloat / cargo-geiger** | 依赖体积与安全性检查 | 免费，每次构建后自动运行 | 免费 |

**决策**：
- 核心开发流：**Hermes 拆解任务 → DeepSeek-TUI 编码 → 人工+Cursor 审查**。
- 仅当遇到 DeepSeek-TUI 处理困难的复杂重构（超过 5 个文件联动）时，手动切换到 Claude Code 辅助。
- 不引入其他本地模型部署，保持工具链极简。

**协作流程**：
```
Hermes 接受开发任务
   ├─ 调用 秋毫mem 获取实时内存数据
   ├─ 将内存约束注入 PRD
   ├─ 将核心逻辑任务转发给 DeepSeek-TUI 编码
   ├─ 将样板/配置/测试任务转发给 Reasonix 编码
   └─ 完成后 → 人工 + Cursor 审查
       完成后 → Hermes 调用 秋毫mem 验证新功能在预期负载下的表现
```

**成本对比**：

| 方案 | 月成本 | 年成本 |
|------|--------|--------|
| V1.0 方案 (Cursor + Claude) | $80-130/月 | $960-1560 |
| **V1.1 方案 (Hermes + DeepSeek-TUI + Cursor)** | **$50-70/月** | **$600-840** |

**成本节省约 40%，且获得 Hermes 的自动化任务调度能力。**

## 二、技术框架选型（无变化）

保持 V1.0 的全部选型不变，与 DeepSeek-TUI 的技术栈完全兼容。

| 层级 | 选型 | 理由 | 禁用项 |
|------|------|------|--------|
| **语言** | Rust stable | 已定，零成本抽象，无 GC，单二进制 | 无 |
| **CLI 框架** | `clap` (derive 模式) | 社区事实标准 | 不用 `argh`、`structopt` |
| **序列化** | `serde` + `serde_json` | 无可替代 | 无 |
| **错误处理** | `thiserror` + `anyhow` | 库层与应用层分离 | 不用 `eyre` |
| **配置文件** | `toml` crate + `serde` | 与配置格式一致 | 不用 `config` crate |
| **平台采集** | Linux: 直接读 `/proc/meminfo` | 零依赖，< 100 行 | 不用 `procfs` |
| | macOS: `libc` + `sysctl` | 比 `vm_stat` 解析更稳定 | 不用 `sysctl` crate |
| **异步运行时** | **不用。完全同步。** | 短生命周期 CLI，避免二进制膨胀 | 禁止 `tokio` |
| **日志** | `eprintln!` 宏 | 极轻量 | 不引入 `log` + `env_logger` |
| **静态链接** | `musl` target (Linux) | 全静态编译，消除 glibc 版本兼容问题 | 无 |

**说明**：DeepSeek-TUI 内部使用 `tokio`，但秋毫mem 作为独立进程，通过 MCP 协议通信，两者异步运行时互不干扰。秋毫mem 的同步架构是正确选择，不需要为协同而修改。

## 三、编程风险点识别（无变化）

V1.1 新增风险点：**DeepSeek-TUI 在跨文件重构时的架构一致性**。已通过"人+Cursor 审查"环节缓解。

| 风险编号 | 风险描述 | 严重程度 | 发生概率 | 缓解措施 |
|----------|----------|----------|----------|----------|
| **R1** | macOS 内存压力计算失准 | **致命** | 高 | 第二周核心攻坚；使用 `memory_pressure` sysctl；在真实 8GB Mac 上验证 |
| **R2** | `/proc/meminfo` 解析兼容性 | 高 | 中 | 键值对解析；未知字段 warn 不 crash |
| **R3** | 依赖膨胀 | 中 | 中 | `cargo-bloat` 每次构建检查 |
| **R4** | `unsafe` 代码风险 | 高 | 低 | 核心逻辑禁止；macOS 封装集中管理 |
| **R5** | 配置优先级覆盖遗漏 | 中 | 低 | 集成测试覆盖全部组合 |
| **R6** | `musl` 静态链接兼容性 | 低 | 极低 | CI 同时构建 `gnu` 备选 |
| **R7 (新增)** | DeepSeek-TUI 多文件重构一致性差 | 中 | 中 | 限制单次任务文件数 ≤ 5；人工+Cursor 审查关键架构 |

## 八、Hermes 与 DeepSeek-TUI 集成配置（新增）

以下为 Hermes 的 MCP 配置示例，将秋毫mem 和 DeepSeek-TUI 注册为可调用工具：

```toml
# Hermes MCP 配置文件
[mcp_servers.hawkeye_mem]
command = "hawk-eye-mem"
args = ["--mcp"]  # 以 MCP Tool 模式运行

[mcp_servers.deepseek_tui]
command = "deepseek-tui"
args = ["--mode", "agent"]  # 以 Agent 模式接受编码任务
```

**工作流示例**：

1. Hermes 接收开发任务："实现 `GuidanceGenerator` 模块"。
2. Hermes 调用 `hawkeye_mem.check_memory`，获取当前可用内存（如 12GB）。
3. Hermes 评估内存充足，将任务拆解为 3 个子任务，其中 2 个简单任务并行交给 `deepseek_tui` 的 Flash 模型，1 个核心逻辑任务交给 Pro 模型。
4. 编码完成后，Hermes 调用 `hawkeye_mem.check_memory` 验证新模块在模拟负载下的内存行为。
5. 人工 + Cursor 审查最终代码，确认架构一致性。
