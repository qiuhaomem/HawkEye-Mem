# 秋毫mem V0.2 · 测试用例文档 多方评审会议纪要

**评审对象**：V0.2 测试用例文档（HM-TC-002）
**会议主持**：测试经理
**参与方**：产品经理、技术负责人、安全专家、PMO
**创始人列席**
**结论性质**：通过，含 6 条整改要求


### 👤 产品经理

测试用例的覆盖广度没问题，但从用户场景角度看，有两个体验相关的验收标准不够具体。

**第一，`--compare` 的"高亮推荐项"验收标准太模糊。** IT-CMP-003 只写了"标注哪个更适合当前环境"，但没有定义什么叫"更适合"。是内存占用更小？还是磁盘空间要求更低？还是综合评分？建议在用例中明确：推荐逻辑 = 所有资源约束全部通过的前提下，内存占用最小的模型。如果多个模型都通过，选内存占用最小的；如果全部不通过，选 gap 最小的。这个规则定了，测试才有明确的预期结果。

**第二，`--list-models` 的颜色判定边界没定义。** UT-ML-007 写了"绿色/红色"，但没有给出具体的判定阈值。建议和 `--can-run` 的 verdict 判定保持一致：可用资源满足模型最低要求 → 绿色；满足但有警告 → 黄色；不满足 → 红色。阈值明确后，测试才能写断言。

**第三，手动验收场景 MA-010（多模型对比）需要补充一个"用户第一次用"的视角。** 当前只验证了功能正确性，但没有验证一个新手能不能看懂输出。建议增加一个子场景：给一个完全没用过秋毫mem 的开发者，让他只看 `--compare` 的输出，问他"你会选哪个模型"，如果他选的不是推荐项，说明输出表达有问题。

**结论**：通过。要求 IT-CMP-003 明确推荐逻辑、UT-ML-007 明确颜色阈值、MA-010 增加新手理解子场景。


### 👤 技术负责人

测试用例和代码实现之间有几处可能不一致的地方，需要提前对齐。

**第一，Agent 进程 CPU 的"无结果"和"检测失败"是两回事。** UT-CPU-008 的预期结果是 `agent_processes_percent` 为 None 或 0，但这两种情况的语义完全不同：None = 无法获取（平台不支持或权限不足），0 = 获取成功但确实没有 Agent 进程在跑。代码实现里这两个路径是分开的，测试用例应该分开验证。建议拆成两个用例：UT-CPU-008A（无可匹配进程 → 0）和 UT-CPU-008B（平台不支持 → None）。

**第二，GPU 止损的"8 工时"是开发流程约束，不是代码行为，不应该放在单元测试里。** UT-GPU-001 的用例名写了"8 工时止损"，但单元测试跑的是代码逻辑，不是开发工时。建议把这个用例的验证目标改为"NVML 方案能正确获取显存数据"，止损条件移到手动验收或 PMO 的进度检查清单里。

**第三，回归测试 REG-005 只验证了配置文件向后兼容的正常路径，缺少边界情况。** 如果 V0.1 的配置文件里有一个 V0.2 不认识的字段，应该静默忽略而不是报错。建议增加 REG-005B：V0.1 配置文件含未知字段，V0.2 加载时正常忽略并输出 warning 到 stderr。

**第四，集成测试 IT-CLI-020 验证了 `--can-run --json` 输出不含 `agent_guidance`，但没有验证 `--json`（不带 `--can-run`）输出不含 `deployment_assessment`。** 这两个字段是互斥的，但只验证了一个方向。建议增加 IT-CLI-020B 做反向验证。

**结论**：通过。要求拆分 Agent CPU 用例、GPU 止损移到手动验收、增加配置文件未知字段回归测试、增加 JSON 输出互斥反向验证。


### 👤 安全专家

技术方案评审时我提的三个安全要求，测试用例已经覆盖了磁盘路径脱敏（UT-DK-011）、NVML unsafe 封装（UT-GPU-010）、nvidia-smi header 匹配（UT-GPU-003）。但有一个遗漏。

**GPU 采集的错误信息可能泄露系统信息。** 当 `nvidia-smi` 不在 PATH 时，UT-GPU-004 预期输出"nvidia-smi not found"。但如果 `nvidia-smi` 存在但执行失败（比如驱动版本不兼容），错误信息里可能包含完整的命令路径、驱动版本号、甚至 GPU 型号。这些信息如果被 Agent 日志记录并分享出去，会暴露硬件配置。建议增加一个用例：当 GPU 采集失败时，错误信息不包含路径、版本号等可定位到具体系统的信息，只输出通用提示。

**结论**：通过。要求增加 GPU 采集失败时的错误信息脱敏用例。


### 👤 PMO

测试用例的数量和覆盖范围可以支撑 V0.2 的质量门禁。但执行计划有三个时间点需要调整。

**第一，回归测试的 8 个用例必须在 W1 重构完成后的当天跑完。** 如果回归测试发现 V0.1 功能被破坏，后续所有开发必须暂停，直到修复。这不是"W1 内完成"，而是"重构完成后 4 小时内完成"。建议在 W1 的时间表里把回归测试标注为重构的阻塞性后置任务。

**第二，手动验收的 12 个场景中，MA-004 到 MA-009 需要特定的硬件状态。** 磁盘临界、CPU 高负载、GPU 有无——这些不是随时都能测的。建议提前锁定测试环境：
- MA-005（磁盘临界）：用 `dd` 创建大文件填充磁盘
- MA-007（CPU 高负载）：用 `stress` 或 `yes` 命令模拟
- MA-008/009（GPU）：需要至少一台有 NVIDIA GPU 的机器，提前预约

**第三，GPU 相关的 11 个单元测试用例（UT-GPU-001~011）中，UT-GPU-001~009 需要在真实 GPU 环境运行。** 如果 CI 中没有 GPU runner，这些用例在 CI 中会全部 skip。建议在测试文档中标注哪些用例需要特殊硬件，并给出 CI 中 skip 时的替代验证方案（如 mock）。

**结论**：通过。要求回归测试标注为阻塞性任务、手动验收设备需求提前锁定、GPU 用例标注硬件依赖。
