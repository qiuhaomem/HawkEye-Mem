## 二、单元测试

### 2.1 数据采集器 (TC-DAT-01 ~ TC-DAT-10)

#### TC-DAT-01：正常解析 state.db
- **前置**：伪造 state.db，含 10 条 sessions 记录，涵盖不同模型
- **输入**：读取 state.db
- **预期**：解析出 10 条 TokenRecord，字段完整
- **断言**：`count == 10`, `model_set.contains("deepseek-v4-flash")`

#### TC-DAT-02：state.db 不存在的降级
- **前置**：state.db 文件缺失
- **输入**：读取 state.db
- **预期**：不报错，返回空记录列表，打印降级信息
- **断言**：`records.is_empty()`，`stderr.contains("降级")`

#### TC-DAT-03：正常解析 agent.log
- **前置**：agent.log 含 10 条标准格式的 API call 日志行
- **输入**：解析 agent.log
- **预期**：提取 10 条 TokenRecord，含 in/out/cache/latency 字段
- **断言**：`count == 10`，`in_tokens > 0`

#### TC-DAT-04：agent.log 格式异常行
- **前置**：agent.log 含 8 条标准行 + 2 条异常行
- **输入**：解析 agent.log
- **预期**：解析出 8 条有效记录，跳过 2 条异常，不中断
- **断言**：`count == 8`

#### TC-DAT-05：agent.log 日志轮转
- **前置**：agent.log 含最后 5 条，agent.log.1 含前 5 条
- **输入**：解析 agent.log（同时读取 log + log.1）
- **预期**：合并 10 条记录，去重
- **断言**：`count == 10`

#### TC-DAT-06：混合数据源优先级
- **前置**：state.db 和 agent.log 均有数据
- **输入**：同时读取两个数据源
- **预期**：优先使用 state.db 数据，agent.log 补充缺失
- **断言**：`source_priority == "state_db"`

#### TC-DAT-07：数据源均不可用
- **前置**：state.db 和 agent.log 均不存在
- **输入**：尝试读取
- **预期**：返回空，提示"无 Hermes Agent 数据"
- **断言**：`records.is_empty()`

#### TC-DAT-08：空 state.db（0 记录）
- **前置**：state.db 存在但 sessions 表为空
- **输入**：读取 state.db
- **预期**：返回空记录列表
- **断言**：`records.is_empty()`，无错误输出

#### TC-DAT-09：state.db schema 版本不兼容
- **前置**：state.db 缺少 input_tokens 列
- **输入**：尝试查询
- **预期**：降级到 agent.log-only 模式
- **断言**：`fallback == true`

#### TC-DAT-10：large state.db（1000+ 记录）
- **前置**：state.db 含 1000 条 sessions 记录
- **输入**：读取全部
- **预期**：5 秒内完成解析
- **断言**：`duration_sec < 5.0`

### 2.2 浪费分析引擎 (TC-ANL-01 ~ TC-ANL-08)

#### TC-ANL-01：冷启动过大检测
- **前置**：首轮 API call input_tokens = 45000
- **输入**：分析 TokenRecords
- **预期**：标记 waste_type = "cold_start"，严重度 = "high"
- **断言**：`waste.cold_start.detected == true`，`waste.cold_start.tokens_wasted >= 15000`

#### TC-ANL-02：缓存命中率低检测
- **前置**：缓存命中率 = 30%（15/50 次命中）
- **输入**：分析 TokenRecords
- **预期**：标记 waste_type = "low_cache_hit"
- **断言**：`waste.low_cache_hit.detected == true`，`waste.low_cache_hit.hit_rate == 30.0`

#### TC-ANL-03：技能冗余检测（模拟）
- **前置**：技能列表有 80 个技能，但只引用了 5 个
- **输入**：读取技能列表配置 + token 注入分布
- **预期**：标记 waste_type = "skill_bloat"
- **断言**：`waste.skill_bloat.detected == true`

#### TC-ANL-04：多浪费场景同时存在
- **前置**：冷启动大 + 缓存低 + memory 臃肿同时满足
- **输入**：分析
- **预期**：输出 3 个 waste 条目，按节省金额排序
- **断言**：`wastes.len() == 3`，第一项节省金额最大

#### TC-ANL-05：无浪费场景
- **前置**：首轮 input=20K，cache_hit=99%，memory=2K
- **输入**：分析
- **预期**：未检测到任何浪费场景
- **断言**：`wastes.is_empty()`

#### TC-ANL-06：冷启动边缘值（29K tokens）
- **前置**：首轮 input_tokens = 29000（阈值 30000）
- **输入**：分析
- **预期**：不标记为冷启动浪费
- **断言**：`waste.cold_start.detected == false`

#### TC-ANL-07：缓存命中率边缘值（49.5%）
- **前置**：缓存命中率 = 49.5%（阈值 50%）
- **输入**：分析
- **预期**：标记为缓存命中率低
- **断言**：`waste.low_cache_hit.detected == true`

#### TC-ANL-08：无数据时分析
- **前置**：TokenRecords 为空
- **输入**：分析
- **预期**：返回空 waste 列表 + 提示"数据不足"
- **断言**：`wastes.is_empty()`，`message.contains("数据不足")`

### 2.3 费用换算器 (TC-CST-01 ~ TC-CST-05)

#### TC-CST-01：内置定价表完整
- **前置**：加载内置定价表
- **输入**：查询所有已注册模型
- **预期**：至少包含 5 个主流模型
- **断言**：`pricing.contains("deepseek/deepseek-v4-flash")`

#### TC-CST-02：已知模型费用计算
- **前置**：100K input + 20K output，deepseek-v4-flash
- **输入**：`calculate_cost("deepseek/deepseek-v4-flash", 100000, 20000)`
- **预期**：费用 = 100000/1_000_000 * 0.07 + 20000/1_000_000 * 0.28 = $0.0126
- **断言**：`cost == 0.0126`

#### TC-CST-03：含缓存命中的费用
- **前置**：100K input（其中 50K 缓存命中）+ 20K output
- **输入**：`calculate_cost_with_cache(...)`
- **预期**：缓存命中部分半价或特定费率
- **断言**：`cost < 0.0126`（纯非缓存）

#### TC-CST-04：未注册模型
- **前置**：模型名不在定价表内
- **输入**：`calculate_cost("unknown-model", 1000, 500)`
- **预期**：费用标记为 "unknown"，不报错
- **断言**：`cost == "unknown"`

#### TC-CST-05：用户自定义定价覆盖
- **前置**：用户自定义定价文件存在，覆盖了 deepseek 的价格
- **输入**：加载定价
- **预期**：用户定价优先级高于内置
- **断言**：`price.input_per_1M == user_defined_value`

## 三、集成测试

### 3.1 CLI --token-budget (TC-CLI-01 ~ TC-CLI-10)

#### TC-CLI-01：--token-budget status 正常输出
- **前置**：正常安装，有 state.db 或 agent.log 数据
- **输入**：`hawk-eye-mem --token-budget status`
- **预期**：输出 Token 消耗总览（总 tokens、费用估算、趋势）
- **断言**：输出包含 "Tokens" 和 "$"

#### TC-CLI-02：--token-budget status --json
- **前置**：同上
- **输入**：`hawk-eye-mem --token-budget status --json`
- **预期**：输出有效 JSON
- **断言**：`jq .` 解析成功，含 total_tokens / estimated_cost 字段

#### TC-CLI-03：--token-budget waste
- **前置**：有浪费场景数据
- **输入**：`hawk-eye-mem --token-budget waste`
- **预期**：输出浪费分析结果
- **断言**：输出含 "冷启动" / "缓存" / "技能" 等关键词之一

#### TC-CLI-04：--token-budget suggest
- **前置**：检测到浪费场景
- **输入**：`hawk-eye-mem --token-budget suggest`
- **预期**：输出优化建议列表（编号 + 描述 + 预期节省）
- **断言**：列表非空，含 "建议" 字段

#### TC-CLI-05：--token-budget suggest --json
- **前置**：有优化建议
- **输入**：`hawk-eye-mem --token-budget suggest --json`
- **预期**：输出 JSON 数组
- **断言**：`jq length > 0`，每项含 id / description / expected_savings

#### TC-CLI-06：--token-budget apply 1 --dry-run
- **前置**：有建议 ID=1
- **输入**：`hawk-eye-mem --token-budget apply 1 --dry-run`
- **预期**：显示 diff，不实际修改文件
- **断言**：输出含 "DRY RUN"，文件内容不变

#### TC-CLI-07：--token-budget apply 1（实际执行）
- **前置**：有建议 ID=1
- **输入**：`hawk-eye-mem --token-budget apply 1`
- **预期**：备份配置 → 修改 → 验证成功
- **断言**：配置被修改，备份文件存在

#### TC-CLI-08：--token-budget apply --rollback
- **前置**：刚刚执行过 apply
- **输入**：`hawk-eye-mem --token-budget apply --rollback`
- **预期**：恢复到最近备份
- **断言**：配置恢复为 apply 前的值

#### TC-CLI-09：--token-budget apply 无效 ID
- **前置**：建议列表只有 3 条
- **输入**：`hawk-eye-mem --token-budget apply 99`
- **预期**：报错，提示有效范围
- **断言**：`stderr.contains("无效的建议ID")`

#### TC-CLI-10：--token-budget 无子命令
- **前置**：无
- **输入**：`hawk-eye-mem --token-budget`
- **预期**：显示帮助信息，列出可用子命令
- **断言**：输出包含 "status", "waste", "suggest", "apply"

## 四、端到端测试

#### TC-E2E-01：全链路——有浪费场景
1. 准备：伪造 state.db（冷启动大 + 缓存低）
2. `hawk-eye-mem --token-budget status` → 输出正常
3. `hawk-eye-mem --token-budget waste` → 检测到浪费
4. `hawk-eye-mem --token-budget suggest` → 给出建议
5. `hawk-eye-mem --token-budget apply 1 --dry-run` → 显示 diff
6. `hawk-eye-mem --token-budget apply 1` → 执行成功
7. 验证：配置已修改，备份存在

#### TC-E2E-02：全链路——无浪费场景
1. 准备：伪造 state.db（首轮 20K，缓存 99%）
2. `hawk-eye-mem --token-budget status` → 输出正常
3. `hawk-eye-mem --token-budget waste` → 未检测到浪费
4. `hawk-eye-mem --token-budget suggest` → 无建议
5. 验证：每一步均正常，无崩溃

#### TC-E2E-03：全链路——无 Hermes 环境
1. 准备：无 state.db / agent.log
2. `hawk-eye-mem --token-budget status` → 提示"无数据"
3. `hawk-eye-mem --token-budget waste` → 提示"无数据"
4. `hawk-eye-mem --token-budget suggest` → 提示"无数据"
5. 验证：每一步均优雅处理，无崩溃

## 五、负向测试

| 编号 | 场景 | 操作 | 预期 |
|------|------|------|------|
| TC-NEG-01 | state.db 权限不足 | 设置 000 权限后读取 | 降级到 agent.log-only，不崩溃 |
| TC-NEG-02 | agent.log 权限不足 | 设置 000 权限后读取 | 降级到 state.db-only，不崩溃 |
| TC-NEG-03 | 磁盘空间不足 | 备份配置时磁盘满 | 报错，提示空间不足 |
| TC-NEG-04 | 配置被意外修改 | 备份文件损坏 | 使用次新备份 |
| TC-NEG-05 | Hermes 配置语法错误 | 读取损坏的 config.yaml | 报错，给出详细错误信息 |
| TC-NEG-06 | 定价表格式异常 | 加载损坏的 custom_prices.json | 跳过自定义定价，使用内置 |
| TC-NEG-07 | 并发冲突 | 同时执行两个 apply | 文件锁保护 |
| TC-NEG-08 | 超大 input tokens 溢出 | input_tokens > 2^31 | 使用 u64，正常处理 |
