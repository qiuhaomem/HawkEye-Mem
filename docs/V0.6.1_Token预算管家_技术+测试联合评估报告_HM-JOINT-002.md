## 二、技术可行性评估

### 2.1 数据源解析

| 数据源 | 格式 | 解析方式 | 难度 | 风险 |
|--------|------|---------|------|------|
| Hermes agent.log | 文本 JSON Lines | `serde_json` 按行解析 | ⭐ | 日志轮转可能丢数据 |
| Hermes state.db | SQLite | `rusqlite` 读取 sessions 表 | ⭐⭐ | Schema 版本兼容（V0.5已验证） |
| 各模型定价表 | 内置 JSON | 编译期嵌入 | ⭐ | 定价变化需更新版本 |
| Hermes config.yaml | YAML | `serde_yaml` 读取/写入 | ⭐⭐ | YAML 注释可能丢失（需用 `--dry-run` + 手动确认） |

**关键决策**：核心分析引擎用 Rust 编译进 CLI 二进制（零依赖），配置修改用 Python Skill 做安全包装（带备份 + 回滚）。

### 2.2 技术实现路径

```
hawk-eye-mem --token-budget status         # 当前Token消耗总览
        ├─ 从 state.db 读取最近24h tokens
        ├─ 从 agent.log 读取最近24h API calls
        ├─ 按模型/会话/时间聚合
        └─ 输出摘要 + 费用估算

hawk-eye-mem --token-budget waste          # 浪费分析
        ├─ 分析每会话的 cache_hit_rate
        ├─ 分析首轮 input_tokens 趋势
        ├─ 识别低效来源（skills冗余/memory臃肿/cold start）
        └─ 输出浪费场景 Top-N + 可节省费用估算

hawk-eye-mem --token-budget suggest        # 优化建议
        ├─ 基于浪费分析生成优化列表
        ├─ 每项含：问题描述 / 建议动作 / 预期节省 / 风险等级
        └─ 支持 --json 输出供 MCP 消费

hawk-eye-mem --token-budget apply <ID>     # 执行优化（带 --dry-run）
        ├─ 备份当前配置
        ├─ 执行优化动作
        ├─ 验证结果
        └─ 回滚（如果失败）
```

### 2.3 关键模块与代码量估算

| 模块 | 语言 | 预估代码量 | 依赖 |
|------|------|-----------|------|
| state.db 解析器 | Rust | ~150 行 | `rusqlite`（可选编译） |
| agent.log 解析器 | Rust | ~120 行 | `serde_json`（已有） |
| 浪费分析引擎 | Rust | ~200 行 | 无 |
| 优化建议生成器 | Rust | ~150 行 | 无 |
| 费用换算器 | Rust | ~80 行 | 编译期定价表 |
| 配置修改执行器 | Rust | ~100 行 | `serde_yaml` |
| CLI `--token-budget` 命令 | Rust | ~50 行 | `clap`（已有） |
| MCP 工具 3 个 | Python | ~100 行 | 无 |
| **总计** | | **~950 行** | 新增依赖：`rusqlite`、`serde_yaml` |

### 2.4 风险与缓解

| 风险 | 概率 | 影响 | 缓解方案 |
|------|:---:|:----:|---------|
| state.db schema 变化 | 中 | 高 | Schema 版本检测，不匹配时降级为 agent.log-only 模式 |
| agent.log 日志轮转丢数据 | 高 | 中 | 同时读最近的 2 个日志文件（`agent.log` + `agent.log.1`） |
| YAML 配置修改破坏注释 | 中 | 中 | 默认 `--dry-run`，应用前显示 diff，支持 `--rollback` |
| 定价表过时 | 中 | 低 | 内置定价 + 用户可配置自定义定价文件 |
| Python MCP 依赖冲突 | 低 | 中 | MCP 脚本用纯 Python 标准库，零外部依赖 |

## 三、测试策略

### 3.1 测试层级

| 层级 | 工具 | 覆盖内容 |
|------|------|---------|
| 单元测试 | Rust `#[cfg(test)]` | 解析器（state.db / agent.log / 定价表）、浪费分析、费用计算 |
| 集成测试 | `tests/` | `--token-budget` 各子命令、MCP 工具调用、配置修改 |
| 端到端测试 | 脚本 | 从真实 agent.log + state.db 生成报告、执行优化并验证 |
| Skill 测试 | Hermes | Token 预算 Skill 加载、MCP 工具调用 |

### 3.2 测试场景矩阵

| 场景 | 输入 | 预期输出 | 优先级 |
|------|------|---------|:------:|
| 正常数据 - 有缓存命中 | state.db 含 7 天数据 | 缓存命中率 > 50% | P0 |
| 正常数据 - 无缓存 | state.db 含数据但 cache=0 | 缓存命中率 = 0%，标记为浪费 | P0 |
| 冷启动过大 | 首轮 input > 30K tokens | 标记"冷启动过大"浪费 | P0 |
| 空数据 - 刚安装 | 无 state.db 或 agent.log | 优雅提示，建议先使用一段时间 | P0 |
| 状态.db 不存在 | 文件缺失 | 降级为 agent.log-only 模式 | P0 |
| 日志格式异常 | 非标准格式的行 | 跳过该行，不中断整体分析 | P1 |
| 费用换算 - 已知模型 | deepseek-v4-flash | 正确换算费用 | P0 |
| 费用换算 - 未知模型 | 自定义模型名 | 使用用户自定义定价或标记"未知" | P1 |
| 优化建议 - 冷启动 | 冷启动过大检测到 | 建议：精简技能列表，减少 input 约 30K tokens | P0 |
| 优化建议 - 缓存命中低 | 缓存命中率 < 50% | 建议：调整缓存 TTL/预取策略 | P0 |
| 优化建议 - 多问题 | 冷启动大 + 缓存低 + memory 臃肿 | 按优先级排序输出 | P1 |
| 配置修改 - dry-run | `--token-budget apply 1 --dry-run` | 显示 diff，不实际修改 | P0 |
| 配置修改 - 实际执行 | `--token-budget apply 1` | 备份配置 → 修改 → 验证 | P0 |
| 配置修改 - 回滚 | 修改后显式回滚 | 恢复到备份 | P0 |
| MCP get_token_budget_status | 无参 | 返回当前 Token 状态 JSON | P0 |
| MCP get_token_budget_suggestions | 无参 | 返回优化建议列表 JSON | P0 |
| MCP run_token_budget_optimization | id + dry_run | 返回优化结果 JSON | P0 |
| 边缘 - 超大日志 | 10000+ 条 API call | 5 秒内完成分析 | P1 |
| 边缘 - 同时多种浪费 | 所有浪费场景同时存在 | 按潜在节省金额排序 | P1 |
| 负向 - 无 Hermes 环境 | 非 Hermes Agent | 提示"仅支持 Hermes Agent" | P0 |
| 负向 - 权限不足 | 无法读 state.db | 提示权限问题 | P0 |
| 定价表 - 内置完全覆盖 | 所有已知模型 | 所有模型都有对应价格 | P1 |
| 定价表 - 缺失部分模型 | 部分模型无价格 | 标记为"unknown_cost"，不影响其他 | P1 |
