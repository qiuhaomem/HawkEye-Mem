# 秋毫mem V0.5 · Token 审计需求调研报告

## 一、调研背景

Agent 框架（如 Hermes）的 Token 消耗对用户来说是一个黑盒：每个月账单来了才知道花了多少钱，但钱花在哪、哪些是必要的、哪些是浪费的——完全不知道。用户普遍存在"API 焦虑"：感觉在烧钱，但没有工具帮他们算账。

Token 审计要解决的就是这个信息不对称。

**用户原声**：
- "我这个月 API 花了 $50，但我不知道哪些对话在烧钱"
- "cron 任务是不是偷偷在用 LLM 烧我的钱？"
- "MCP 重试了 1436 次，我完全不知道"
- "我以为烧了很多钱，其实大部分是必要的内部开销"


## 二、用户痛点

| 痛点 | 描述 | 频次 |
|------|------|------|
| 总账不清 | 不知道总共花了多少 Token、多少钱 | 高频 |
| 来源不明 | 不知道是微信对话在烧钱还是 cron 在烧钱 | 高频 |
| 浪费不可见 | 失败重试、429 限流、MCP 连不上等浪费完全不知道 | 中频 |
| 成本归属模糊 | 不知道哪些 Agent/任务是大头 | 中频 |
| 优化无方向 | 知道花了钱，但不知道怎么优化 | 中频 |


## 三、数据源分析

| 数据源 | 位置 | 包含信息 | 可审计维度 |
|--------|------|---------|-----------|
| Hermes state.db | `~/.hermes/state.db` | 会话记录、消息数、token 数、费用 | 总账、来源分布、会话排行 |
| agent.log | `~/.hermes/agent.log` | 错误日志、重试记录、MCP 调用状态 | 浪费检测、429 统计、MCP 故障 |
| crontab | 系统 crontab | cron 任务列表、执行时间 | cron 审计 |
| Hermes 配置文件 | `~/.hermes/config.yaml` | Provider 配置、模型选择 | 定价参考 |


## 四、竞品分析

**结论先行：没有成熟的 Token 审计产品。所有竞品都是"只干一件事"的散装工具。秋毫mem是第一个把五维审计拉通的。**

### 4.4 秋毫mem的核心竞争力

**1. 维度最全，竞品都是"只干一件事"**

现有的每个工具都只解决Token审计的一个切面：tokenly告诉你花了多少钱，tokscale告诉你哪里浪费了，rasad-ai告诉你Agent在干什么——但没有一个把五维（总账、分布、浪费、cron、费用对比）全部拉通。**秋毫mem是第一个。**

**2. "一句话总结"是杀招**

所有竞品的输出都是数据、表格、Dashboard——用户需要自己分析。秋毫mem直接给结论：`📊 总消耗 199M tokens = $4.21 | 真实浪费 1.8M (0.9%) = $0.27 | 真相：大部分是必要开销`。用户截图传播这一句话，比传播任何Dashboard截图都有效。

**3. "来源分布"打的是差异化盲区**

绝大多数竞品统计的是"哪个模型花了多少钱"、"哪个项目花了多少钱"，但没有任何一个工具回答："我的Token是微信对话在烧钱，还是cron任务在烧钱，还是子Agent在烧？"这是真实的用户问题——用户在Hermes里既跑微信对话、又跑定时任务、又跑子Agent，但账单是一团乱麻。秋毫mem是第一个拆解这个问题的。

**4. "cron审计"是独占空白**

所有竞品**没有一家**做cron审计。我们调研了17个工具，0个支持。这个功能的开发成本极低（就是解析crontab + 匹配state.db中的时间戳），但用户价值巨大——"哪个定时任务在偷偷用LLM烧我的钱？"是每个重度Hermes用户的真实恐惧。

**5. 双饵协同，竞品抄不了**

tokenly做不了浪费检测，tokscale做不了总账，rasad-ai做不了cron审计。但秋毫mem的Token审计 + 缓存策略是联动的：审计报告里展示"缓存策略帮你省了X美元"，缓存报告里引导"想查总账？运行Token审计"。**两个Skill互相导流，共同依赖秋毫mem底座。** 竞品即使抄了一个功能，也抄不走这个生态闭环。

**6. 本地优先 + Hermes原生集成**

大部分竞品面向AI编码工具（Claude Code、Codex），没有一个原生支持Hermes的state.db和agent.log。我们的用户群（Hermes Agent用户）是这些竞品完全没覆盖的空白地带。同时，所有数据处理在本地完成，不上传任何数据，这是和modelstat等云端方案的本质差异。

### 4.5 竞品威胁评估

| 威胁 | 可能性 | 应对 |
|------|--------|------|
| tokenly加审计维度 | 中（6-12个月） | tokenly定位轻量追踪，加五维审计会变重，偏离初心 |
| rasad-ai加浪费检测 | 中（3-6个月） | rasad-ai聚焦编码场景，Hermes用户不是其目标 |
| AgentSight开放给Hermes | 低（阿里云生态绑定） | 暂时不担心 |
| 新项目复制我们 | 高（想法可抄） | **先发优势 + 双饵生态 + Hermes独占** 构成壁垒 |

**最大威胁不是竞品，是"没人知道秋毫mem有Token审计"。** 所以推广策略和钓鱼叙事至关重要。

### 4.6 竞品分析结论

**答案是：还没有成熟的Token审计产品。所有竞品都是"只干一件事"的散装工具。秋毫mem是第一个把五维审计拉通的。** 但窗口期有限——6个月后rasad-ai或tokscale就可能补齐短板。

**我们的护城河**不是代码（Token审计的代码量不大），是**生态闭环**：缓存策略 + Token审计双饵互相导流，秋毫mem底座为两者提供决策依据，Hermes用户是所有这些竞品都未覆盖的独占人群。

**但有一个硬伤必须补上**：缓存策略对不同模型的兼容性完全不同，如果用户在Groq上用了不支持的模型，99.7%的命中率就会崩塌。V0.5的Skill需要在启动时自动检测当前模型是否支持Prefix Caching，不支持时给明确提示，否则用户会觉得"这工具骗人的"。

## 七、产品经理总结

V0.5 现在有两个鱼饵了。缓存策略负责"帮你省钱"，Token 审计负责"帮你查账"。用户因为想省钱装了缓存策略，发现还能查账——两个 Skill 互相导流，秋毫mem 在两个 Skill 中都是底层渔具。

Token 审计的数据源是 Hermes 的 state.db 和 agent.log，所有处理本地完成，不上传任何数据。报告的核心是一句话总结 + 结构化明细 + 浪费检测 + 费用对比。

竞品分析扫了14个工具/项目，核心结论：**没有一家做全五维（总账/分布/浪费/cron/对比），cron审计是100%独占空白。窗口期预计6个月。** 最大威胁不是竞品复制，是没人知道秋毫mem有这功能——推广 = 护城河。

**竞品分析中发现的一个硬伤**：缓存策略对不同模型的Prefix Caching支持情况差异巨大，Skill启动时必须自动检测当前模型是否支持，不支持时要给明确提示，否则用户遇到不支持的模型会认为"工具骗人的"。
