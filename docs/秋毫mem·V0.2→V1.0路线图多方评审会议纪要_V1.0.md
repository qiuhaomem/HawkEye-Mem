# 秋毫mem · V0.2→V1.0 路线图 多方评审会议纪要

**评审对象**：产品路线图 V1.0（HM-ROADMAP-001）
**会议主持**：PMO
**参与方**：产品经理、技术负责人、市场运营、法务、财务、创始人（列席）
**结论性质**：有条件通过，5个决策项中4个已有明确建议，1个需创始人拍板


### 👤 产品经理

路线图的大方向我没意见，但有一个用户体验的坑必须提前填。

**V0.2 的 `--assess` 命令名有问题。** 开发者习惯 `--help`、`--version`、`--json` 都是短参数。`--assess` 拼写容易出错，而且语义不够直白。我建议改为 `--can-run`，用户敲 `hawk-eye-mem --can-run --model llama3-8b`，比 `--assess` 直观得多。参数名本身就是最好的文档。

**第二个问题**：V0.3的"动态校准"有信用风险。Agent跑了好几天，`confidence` 终于从 `conservative` 变成了 `calibrated`，用户以为精准了，结果某天还是崩了。产品上必须给这个"精准"加一个持续的免责提示——即使 `calibrated` 了，也不等于100%准确。我建议 `calibrated` 之后输出里仍然保留一条简短的 `_note`，只是措辞比 `conservative` 阶段温和一些。

**第三个问题**：V0.2的模型参数库如果只预置8个模型，可能会让用户觉得"支持的模型太少"。我们需要在产品文档里明确标注这个参数库是社区共建的，不是官方认证的，并且给出如何贡献新模型参数的指南，把"少"变成"邀请参与"。

**结论**：同意路线图。要求 `--assess` 改名 `--can-run`，校准后的免责提示保留。


### 👤 技术负责人

产品经理的想法在工程上都能落地，但我要指出三个技术风险点。

**GPU NVML绑定的工期是最不确定的。** NVML的Rust绑定目前有两个社区crate：`nvml-wrapper` 和 `nvidia-ml-sys`。前者维护较好但文档稀疏，后者更底层但需要手写大量unsafe。实际编码工期可能比预估多1-2周。建议在W4做一个早期验证：用最简代码拿到显存总量和已用量，如果这个能在一天内跑通，后续就稳了。如果不行，果断切换方案为解析 `nvidia-smi --query-gpu` 的CSV输出，牺牲性能换确定性。

**第二个风险**：V0.3的动态校准需要本地存储历史数据。这会引入第一个非易失性依赖——当前V0.1是无状态的，重启就忘。存储方案需要提前定下来：用SQLite太重，用JSON文件会有并发写问题。我建议用最简单的append-only CSV文件，格式一行一条记录，写在 `~/.config/hawk-eye-mem/calibration.csv` 里。零依赖，人类可读，Agent崩溃也不会损坏数据。

**第三个问题不是风险，是架构提醒**：V0.2同时加了磁盘、CPU、GPU三个Collector，V0.3还要加温度和网络，`MemoryCollector` trait的命名已经不能准确描述它的职责了。V0.2启动时应该把它重命名为 `ResourceCollector`，并让所有采集器实现同一个trait。这只是一次重构，不影响外部API，但必须在V0.2早期做，否则后面改动成本更高。

**结论**：同意路线图。要求V0.2 W4做GPU早期验证，V0.2启动时重命名trait，V0.3的校准存储用CSV。


### 👤 市场运营

路线图的叙事结构很完整，但少了两个关键动作。

**第一，V0.2发布时必须有对比页面。** 不是对比竞品，是对比秋毫mem V0.1和V0.2。用一张图展示：左边V0.1只告诉你内存快炸了，右边V0.2告诉你部署前能不能跑、差多少、怎么调。这个对比本身就是传播素材——告诉社区我们不是在堆功能，而是在补全"Agent环境感知"这件事的完整闭环。

**第二，V0.3/0.4的"动态校准"是一个需要用户见证才能相信的功能。** 靠文字说不清楚。需要录制一个15秒的时间流逝视频：Agent第一次被部署，秋毫mem说 `confidence: "conservative"`，估算4096 tokens；Agent跑了几天，秋毫mem自动校准，`confidence` 变成 `"calibrated"`，估算升到了5500 tokens。这个视频比任何博客都有说服力。

**第三，V1.0的"Web仪表盘"如果只是另一个Grafana，就浪费了这个项目积累的品牌势能。** 我建议仪表盘的视觉风格延续"Agent的家"这个隐喻——不是冷冰冰的监控大屏，而是像一个智能家居控制面板，每个Agent是一个住户，每项资源是房间的容量。这个设计方向如果成立，仪表盘本身就会成为传播事件。

**结论**：同意路线图。要求每个版本发布时有对比页面，V0.3配校准时间流逝视频，V1.0仪表盘设计探索"Agent的家"视觉风格。


### 👤 法务

路线图涉及两个新的法务风险点，需要在对应版本发布前处理。

**V0.2的 `--can-run` 建议有新的责任敞口。** V0.1的建议是"运行时降低上下文"，这个建议即使错了，后果是OOM，而OOM的风险我们已经在DISCLAIMER中覆盖了。但V0.2新增的建议是"可以部署"——如果用户因为秋毫mem说可以部署，下载了50GB的模型，花了三个小时，结果跑不起来，他损失的不只是时间，还有带宽成本。这个场景的损失比OOM更具体、更可量化、更容易引发索赔。需要在V0.2的DISCLAIMER中增加一条明确针对部署评估的免责条款。

**V0.4的远程采集功能如果读取远端机器的系统信息，在中国《个人信息保护法》下，系统资源指标（如内存、CPU使用率）通常不构成个人信息。** 但如果远程采集的配置中包含了主机名、用户名、IP地址等可关联到具体个人的信息，就需要评估是否触发合规义务。建议在技术方案设计时，明确远程采集的数据范围，排除任何可能关联到自然人的标识符。同时提供一份单独的《远程采集数据处理说明》，说明采集了哪些字段、存储在哪里、如何删除。

**结论**：同意路线图。要求V0.2发布前补充部署评估免责条款，V0.4技术方案中明确远程采集数据范围并配套处理说明。


### 👤 财务

路线图涉及的预算在可接受范围内，但有一个商业假设需要验证。

**V0.4的远程采集商业化试点，假设是"有企业愿意为多机监控付费"。** 这个假设在V0.2发布后就应该开始验证，而不是等到V0.4开发完了再去看有没有人买。我建议在V0.2的README底部加一个简短的两行调查："如果你需要监控多台机器的Agent资源状态，请在这个Issue下+1。"社区如果有超过20个企业在Issue下回应，就说明这个需求真实存在，V0.4的商业化投入风险大幅降低。如果几乎没人回应，V0.4的商业化试点应该重新评估。

**第二个提醒**：V1.0的Web仪表盘如果自己开发，需要一个前端开发者，预算会比之前所有版本加起来都高。如果用Grafana等现有方案集成，成本趋近于零但差异化消失。这个决策不需要现在做，但在V0.4结束时必须有明确结论。建议在路线图中标记这个决策的时间窗口。

**结论**：同意路线图。要求V0.2发布时嵌入商业化验证信号，V0.4结束时必须有V1.0仪表盘的技术选型结论。
