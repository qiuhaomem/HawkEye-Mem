# 秋毫mem V0.5 · Token 审计联合评估 多方评审会议纪要

## 评审背景

V0.5 核心功能（缓存策略 Skill + 秋毫mem CacheAdvisor）已开发完成，Token 审计为临时插入需求。评审 HM-JOINT-001 的技术可行性、测试覆盖和双饵协同方案。


## 👤 PMO 发言

先说结论：**这个临时插入需求的时机非常尴尬，但并非不可执行。**

V0.5 核心功能已经开发完成。现在插入 Token 审计，意味着：

1. **不涉及秋毫mem 核心代码改动**：Token 审计是纯 Python Skill，只读 Hermes 的数据，不碰秋毫mem 的 Rust 代码。这一点救了命——不需要重新编译、重新测试秋毫mem 本体。
2. **不影响已完成的缓存策略 Skill**：两个 Skill 独立运行，只在"双饵联动"点有弱耦合（审计报告里引用缓存节省数据、缓存报告底部引导审计入口）。这个联动点是文案级别的，不是代码级别的。

**止损线**：
- Token 审计作为一个**独立 Skill** 开发，不和缓存策略 Skill 合并代码，降低耦合风险
- 如果 W2 结束前 Token 审计未通过集成测试，**立即降级为"V0.5 缓存策略 Skill 的内置报告模块"**，只输出一句话总结 + 基本统计，不做浪费检测和 cron 审计
- V0.5 的发布日期不变——Token 审计通过就双饵齐发，不通过就只发缓存策略，Token 审计作为 V0.5.1 追加

联合评估报告里写"Token 审计可由同一人异步开发或分拆给第二位开发者"。现在已经是临时插入，没有第二位开发者。建议技术负责人直接主导缓存策略的收尾，Token 审计由你本人异步开发。两个模块的代码完全不重叠，上下文切换成本低。

**结论**：通过。要求 Token 审计作为独立 Skill、W2 设定降级止损线、V0.5 发布日期不变。


## 👤 产品经理 发言

联合评估报告的技术部分没问题，但从用户体验角度，有几个文案和流程细节必须补上。

**一句话总结的文案需要多版本测试。** 不同用户的"真相"不同：有的用户浪费占比 50%（"你的浪费比实际使用还多！"），有的用户几乎没浪费。**一句话总结需要根据实际情况动态生成**，而不是一个固定模板。

建议设计 5 种一句话总结模板，覆盖：
1. 浪费极低（<5%）→ 安慰型："大部分是必要开销"
2. 浪费中等（5-20%）→ 提醒型："有 X% 的浪费可以优化"
3. 浪费很高（>20%）→ 警告型："超过 1/5 的 Token 被浪费，建议立即检查"
4. 无 state.db → 引导型："未找到 Hermes 数据库，请确认路径"
5. 数据不足 → 建议型："数据量不足，建议运行一段时间后再审计"

**结构化报告的 emoji 使用需要一致。** 建议在 iTerm2/Terminal.app/VS Code Terminal 三个主流终端上测试 emoji 显示效果，确保不出现方框或乱码。

**双饵联动引导需要克制。** 每个报告底部最多一个引导，且只在首次使用时出现，后续报告不重复引导。

**结论**：通过。要求一句话总结动态生成（5 种模板）、emoji 三终端兼容测试、双饵引导首次后静默。


## 👤 市场运营 发言

Token 审计作为第二个鱼饵，传播价值甚至比缓存策略更高。缓存策略是"省钱"，Token 审计是"查账"——查账天然带有"揭秘"和"真相"的传播属性。

但联合评估报告里**没有提及推广物料**。建议在发布前准备：
- 一张 Token 审计报告的截图（真实数据脱敏后）
- 一条 30 字的传播文案："我一直以为 API 花了很多钱，直到我查了账。真相是：真正浪费的只有 0.9%。"
- 一个 Reddit/HN 帖子标题：《I audited my AI agent's token usage. The truth surprised me.》

Token 审计的"一句话总结"本身就是传播素材。用户会截图那一句话发社交媒体——截图里秋毫mem 的名字就在上面。这和缓存策略的成本报告水印是同一个传播逻辑。

**结论**：通过。要求 Token 审计同步准备推广物料（截图+文案+帖子标题）。


## 👤 法务 发言

Token 审计涉及读取 Hermes 的 state.db 和 agent.log，这两个文件包含用户与 AI 的对话数据。虽然审计报告不输出对话内容，但解析过程中可能意外泄露敏感信息。

**SQL 查询必须只提取聚合数据。** 不允许使用 `SELECT * FROM messages`，必须使用 `SELECT COUNT(*), SUM(prompt_tokens), SUM(completion_tokens), SUM(cost_usd) FROM messages GROUP BY source` 这类聚合查询。代码中需要硬编码白名单字段，不允许动态拼接 SELECT 列。

**agent.log 解析的正则匹配只匹配错误模式，不提取消息内容。** 正则表达式需要在代码评审中单独审查。

**需要更新 DISCLAIMER**。增加一条：Token 审计功能读取 Hermes 的本地数据，审计过程完全在本地完成，不上传任何数据。

**结论**：通过。要求 SQL 查询只做聚合、正则匹配只提错误模式不提取内容、更新 DISCLAIMER。
