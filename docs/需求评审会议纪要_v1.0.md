# 秋毫mem · 需求评审会议纪要

**会议主题**：五级需求分解 V1.0 评审  
**主持人**：高级产品经理  
**参与者**：创始人（列席）、技术负责人（需求方）  
**结论性质**：以下问题需全部确认或整改后方可进入开发。

---

## 议题一：`confidence: "low"` 到底有没有用？

**产品经理提问**：
我们设计了 `confidence` 字段，当用户未配置模型参数时强制标为 `low`。我的问题是：**一个 Agent 拿到 `confidence: "low"` 的建议后，它能怎么办？**

如果 Agent 的逻辑是"置信度低就不采纳"，那秋毫mem 在没有配置文件的情况下，**等于一个废工具**。用户第一次用，没配配置文件，Agent 说"不可信，我忽略"，然后该 OOM 还是 OOM。这就失去了我们"开箱即用"的价值。

**技术负责人回应**：
`low` 的设计初衷是保守——我们宁可不被信任，也不能给出错误建议。

**产品经理裁决**：
这不是二选一。我们要的是**分级可用性，而不是二元信任**。

**整改要求**：
1. `confidence` 从三个值改为两个值：`"conservative"`（未配置）和 `"calibrated"`（已配置/动态校准）。
2. 即使 `confidence: "conservative"`，Agent 仍可采纳建议，但安全边际必须大到足以覆盖常见场景。当前 30% 边际继续保留。
3. 在 JSON 输出的 `agent_guidance` 中新增一个字段 `suggestion`，当 `confidence` 为 `conservative` 时，其值固定为：`"Consider configuring your model parameters for more precise memory estimation. Run 'hawk-eye-mem --init-config' to get started."`。这是给 Agent 和人类共同的引导。

## 议题二：`stop_and_dump` 真的能执行吗？

**产品经理提问**：
当压力水位达到 `critical` 时，我们建议 Agent 执行 `stop_and_dump`，即保存状态并停止推理。但问题是：**如果内存已经耗尽到临界点，Agent 本身还有能力执行"保存状态"这个动作吗？**

这就像告诉一个快淹死的人"你先写封遗书再沉下去"。逻辑上可能成立，物理上未必可行。

**技术负责人回应**：
`critical` 的判断阈值是 `available < 1GB` 或 `used > 95%`。此时操作系统可能已经开始交换，Agent 的进程未必能正常调度。确实存在建议无法被执行的风险。

**产品经理裁决**：
我们的建议必须与物理可行性匹配。

**整改要求**：
1. `stop_and_dump` 改名为 `abort_safely`，语义上更诚实。
2. `abort_safely` 的触发阈值必须更保守：不是等到内存快没了才说，而是 `available < 2GB` 或 `used > 92%` 时就触发。给 Agent 留出真正执行保存操作的内存余量。
3. 在 `reason` 字段中，必须明确告知风险紧迫程度，例如：`"Critical memory pressure detected. Immediately save your work and terminate inference to prevent data loss. Estimated safe buffer: 2GB."`

## 议题三：`--interval` 的 JSON Lines 输出，Agent 真能消费吗？

**产品经理提问**：
我们设计了 `--json --interval 5` 输出 JSON Lines。我承认这对人类开发者监控终端流很友好。但 **Agent 调用 MCP Tool 是请求-响应模式，不是流式消费模式**。

如果我们把这个模式推给 Agent，Agent 会怎么做？它拿到第一行 JSON，处理，然后呢？剩下的行会堆积在缓冲区，或者直接被丢弃。这个功能对 Agent 端是无效设计。

**技术负责人回应**：
这是实话。MCP 的 Tool 调用是单次返回。JSON Lines 确实主要面向人类和脚本管道，不是 Agent。

**产品经理裁决**：
功能保留，但职责边界要划清。

**整改要求**：
1. 在文档和 Skill 示例中，明确标注：**Agent 集成仅使用单次调用模式（`--json` 无 interval）**。
2. `--interval` 模式定位为"开发者调试与脚本集成"功能，不作为 Agent 集成的推荐方式。
3. MCP Tool 定义中，`check_memory` 不接受 `interval` 参数。Agent 需要持续监控时，应由 Agent 自身的循环逻辑多次调用 Tool，而非由秋毫mem 内部循环。

## 议题四：配置文件路径在容器/非标环境下是否可靠？

**产品经理提问**：
我们默认配置路径是 `$HOME/.config/hawk-eye-mem/config.toml`。这在标准桌面 Linux 上没问题。但我们的目标用户可能把 Agent 跑在 Docker 容器里、无头服务器上、或者通过 systemd 启动。

这些环境下 `$HOME` 可能没设置，`$HOME/.config` 可能不存在，甚至整个文件系统是只读的。

**技术负责人回应**：
当前设计是硬编码路径，如果目录不存在，静默跳过。但没有提供覆盖机制。

**产品经理裁决**：
需要一个显式的覆盖入口。

**整改要求**：
1. 新增环境变量 `HAWKEYE_MEM_CONFIG`，优先级高于默认路径。
2. 新增 CLI 参数 `--config <path>`，优先级高于环境变量。
3. 查找优先级（从高到低）：`--config` → `HAWKEYE_MEM_CONFIG` → `$HOME/.config/hawk-eye-mem/config.toml`。
4. 所有路径查找失败时，静默回退默认值，不报错。

## 议题五：`--metric` 的极简输出足够安全吗？

**产品经理提问**：
我们设计了 `--metric available_mb` 输出纯数字 `5800`，方便脚本用 `$()` 捕获。但如果用户写脚本时，秋毫mem 因为权限问题执行失败，返回码是非零，但 stdout 可能是空的或者有错误信息。

脚本里 `RESULT=$(hawk-eye-mem --metric available_mb)` 拿到的可能是空字符串或错误文本，然后直接用于计算，就会产生静默错误。

**技术负责人回应**：
确实。极简输出对错误处理不友好。stderr 和 stdout 的分离依赖于开发者是否正确重定向。

**产品经理裁决**：
极简模式必须保证"输出格式可预测"。

**整改要求**：
1. `--metric` 模式在成功时，stdout 严格输出纯值（数字或字符串），无额外换行以外的内容。
2. `--metric` 模式在失败时，stdout 输出空字符串，stderr 输出错误信息，返回码为非零。
3. 在文档中给出安全的 Shell 调用示例：
   ```bash
   RESULT=$(hawk-eye-mem --metric available_mb)
   if [ $? -ne 0 ]; then
       echo "Error: $RESULT" >&2
       exit 1
   fi
   ```

## 议题六：首次运行引导与免责声明是否冲突？

**产品经理提问**：
我们设计了两件事：首次运行输出"快速引导"，以及单独的 `DISCLAIMER.md` 文件。但用户第一次运行某个新工具时，最想看到的是它能不能用，而不是免责声明。

如果我们把免责声明和快速引导分开，用户大概率只读引导，不看免责声明。这对法务来说是不可接受的。

**技术负责人回应**：
法务要求免责声明必须"显著"。如果它藏在另一个文件里，法务那边过不去。

**产品经理裁决**：
两者必须融合，且免责声明优先。

**整改要求**：
1. 首次运行时，**先**输出简洁版免责声明，**再**输出快速引导。
2. 免责声明控制在三行以内：
   > **NOTICE**: HawkEye Mem provides estimated suggestions only. No warranty of accuracy. Use at your own risk. See DISCLAIMER.md for full terms.
3. 首次运行检测机制：在 `$HOME/.config/hawk-eye-mem/` 下创建一个 `.onboarded` 标记文件。文件存在则跳过引导和免责声明输出。文件不存在则输出，然后创建标记文件。
