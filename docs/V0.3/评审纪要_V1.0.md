# 秋毫mem V0.3 · 需求清单 多方评审会议纪要

**评审对象**：V0.3 需求调研报告 + 五级需求分解表
**会议主持**：高级产品经理
**参与方**：技术负责人、市场运营、法务、安全专家、PMO
**创始人列席**
**结论性质**：通过，含 8 条整改要求


## 👤 技术负责人

需求整体可落地，但动态校准引擎有几个实现层面的问题需要提前明确。

**第一，校准数据的"推理前/后快照"依赖 Agent 主动配合。** 需求里写了 Agent 在推理前后调用秋毫mem，并传入 `tokens_processed`。但这不是秋毫mem 自己能控制的事——Agent 框架需要改造才能传这个参数。建议在 MCP Tool 接口中增加一个可选参数 `tokens_processed`，并在文档中给出 Hermes 等框架的集成示例。如果 Agent 不传这个参数，秋毫mem 无法计算 `bytes_per_token`，校准功能静默跳过。

**第二，CSV 存储的并发写入问题。** 需求说"追加写入 CSV"，但如果两个 Agent 进程同时调用秋毫mem，可能出现写冲突。建议用文件锁（`flock`）保护写入操作。锁获取失败时，放弃本次写入，不阻塞主流程。校准数据丢失一条可以接受，但采集流程不能因为锁等待而超时。

**第三，AMD GPU 采集依赖 `rocm-smi` 命令行工具。** 这个工具在 ROCm 安装后默认在 PATH 中，但路径可能因发行版不同而变化。建议在配置文件中增加 `rocm_smi_path` 字段，允许用户指定完整路径。Apple Silicon 的 Metal API 采集也有类似问题——需要提前验证 `metal-rs` 或 `objc` 绑定在纯命令行环境（非 Xcode 项目）中的可用性。

**第四，连续监控状态机的"连续 N 次"在不同 `--interval` 下意义不同。** `--interval 1` 时 3 次是 3 秒，`--interval 60` 时 3 次是 3 分钟。建议阈值与时间挂钩而非次数：例如"pressure ≥ high 持续 10 秒"触发 warning，而非固定 3 次。这样在不同采集频率下行为一致。

**结论**：通过。要求 MCP Tool 增加 `tokens_processed` 可选参数、CSV 写入加文件锁、AMD 采集支持自定义路径、状态机阈值改为时间挂钩。


## 👤 安全专家

动态校准引入了本地数据存储，这是秋毫mem 第一次在用户机器上"记住"信息。需要关注隐私和数据安全。

**第一，校准 CSV 文件包含模型名和时间戳。** 如果用户跑了一个自己微调的私密模型，模型名可能包含敏感信息（比如项目代号、公司名称）。建议在 CSV 存储时对模型名做一次哈希处理，只保留前 8 位哈希值用于分组，不存明文模型名。同时在 `--calibration-stats` 输出中，模型名也只显示哈希值。

**第二，进程检测遍历 `/proc` 或调用 `sysctl` 是正常操作，但需要明确不读取进程的命令行参数。** 命令行参数可能包含密码、token 等敏感信息。Agent 进程检测只取进程名和内存占用，不碰 `cmdline`。

**第三，温度采集在 Linux 上需要读取 `/sys/class/thermal/`，这个路径在部分容器或虚拟化环境中不存在。** 不存在时静默跳过即可，不需要特殊处理。但建议在文档中说明哪些环境限制会导致温度数据不可用。

**结论**：通过。要求 CSV 中模型名哈希处理、进程检测不读取 cmdline、温度采集不可用时静默跳过并在文档中说明。


## 👤 市场运营

V0.3 的"动态校准"是一个很好的叙事点，但从传播角度看，有两个体验需要优先保证。

**第一，"从 conservative 到 calibrated"这个变化过程，用户应该是"突然发现"的，而不是"一直盯着看"的。** 建议在 Agent 第一次收到 `confidence: "calibrated"` 时，秋毫mem 的输出中附带一条简短的庆祝性提示："你的 Agent 已经收集了足够的数据，现在估算更精准了。"这会让用户觉得"这东西真的在学"，而不是一个冷冰冰的字段变化。这个小细节本身就有传播价值——用户会截图发社交媒体。

**第二，`--calibration-stats` 的输出需要有"成就感"。** 不只是冷冰冰的数字，而是一个小型的学习报告：学了多少次、精度提升了多少、从什么时候开始校准的。这会让用户觉得"我和我的 Agent 一起成长了"。这个情感钩子比任何技术指标都更能打动人。

**结论**：通过。要求 `confidence` 首次升级时输出庆祝性提示、`--calibration-stats` 输出增加叙事性元素。


## 👤 法务

动态校准功能的免责声明需要更新。

**第一，校准数据的准确性声明。** 当前 DISCLAIMER 覆盖了"估算可能不准"。但 V0.3 的校准功能暗示"越用越准"，用户可能因此更加信赖输出，一旦出错反而更愤怒。需要在 DISCLAIMER 中增加一条："动态校准功能通过积累历史数据提高估算精度，但校准后的估算仍可能受系统状态变化、硬件老化、并发负载等因素影响而产生偏差。校准不等于精确测量。"

**第二，校准数据的隐私说明。** CSV 文件存储在用户本地，不上传。但需要在文档中明确告知用户校准数据存储的位置、内容、以及如何删除（`--reset-calibration`）。这既是合规要求，也是建立信任的好机会。

**第三次件，进程检测的合规确认。** 检测同机其他 Agent 进程属于正常的系统监控行为，不涉及个人信息的收集或处理，在国内外隐私法规下均不构成合规风险。但需在文档中说明该功能仅读取进程名和资源占用，不读取命令行参数或其他可能包含个人信息的数据。

**结论**：通过。要求 DISCLAIMER 增加校准准确性声明、文档增加校准数据隐私说明、进程检测合规确认通过。


## 👤 PMO

V0.3 的功能数量比 V0.2 更多，工期预估需要重新评估。

**第一，动态校准引擎是 V0.3 最核心也最耗时的模块。** 它涉及：MCP Tool 接口改造、CSV 存储、校准算法、confidence 升级/降级逻辑、多模型隔离。预估至少 3 周。连续监控状态机相对独立，可以和校准引擎并行开发。

**第二，GPU 完整支持从"实验性"升级为"正式功能"，需要大量测试。** NVIDIA 增强 + AMD 新增 + Apple Silicon 新增，三个平台需要在真实硬件上验证。建议在需求文档中明确：AMD 和 Apple Silicon 的 GPU 采集标记为"正式但需社区反馈"——如果没有对应硬件的测试环境，功能可用但暂不承诺稳定性。

**第三，温度和多 Agent 是 P1 需求，不应该阻塞 P0 的交付。** 建议在 W4 评估：如果 P0 模块（校准+状态机+GPU）已经占满工期，温度和多 Agent 只做最简实现（只采集不预警），完整功能延后到 V0.4。

**结论**：通过。要求校准引擎预留 3 周、AMD/Apple GPU 标记"需社区反馈"、W4 评估是否裁剪 P1 功能。

## ✅ 评审结论

**高级产品经理最终裁定**：V0.3 需求清单通过，准予进入技术方案设计阶段。

**前置条件**：CR-01、CR-02、CR-04、CR-05、CR-06、CR-09、CR-10 在技术方案评审前必须落地。CR-03、CR-07、CR-08 在技术方案中体现。

**修订后的优先级**：

| 优先级 | 内容 |
|--------|------|
| P0 | 动态校准引擎（含 MCP 改造 + CSV 存储 + 算法）、连续监控状态机（时间挂钩）、GPU 完整支持（NVIDIA 增强 + AMD + Apple Silicon） |
| P1 | 温度/功耗监控（最简实现：只采集不预警）、多 Agent 基础检测 |
| P2 | `--calibration-stats` / `--reset-calibration` / `--gpu-list` CLI |
| 延后 | 温度墙预警完整版 → V0.4、多 Agent 协调建议 → V0.4 |
