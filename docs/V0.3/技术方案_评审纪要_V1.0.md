## 👤 架构师

方案整体架构演进合理，但有几个设计点需要进一步打磨，否则会给 V0.4/V1.0 埋坑。

**第一，`CsvStore` 的职责过于单一，且直接硬编码了 CSV 格式。** 未来 V0.4 的环境指纹和历史趋势需要更复杂的查询（如按时间范围聚合、跨模型统计）。建议在 V0.3 就将存储层抽象为一个 `CalibrationStore` trait，`CsvStore` 作为其默认实现。这样 V0.4 替换为 SQLite 或 JSON 存储时，上层校准引擎无需改动。

```rust
pub trait CalibrationStore {
    fn append(&self, point: CalibrationPoint, model_hash: &str) -> Result<()>;
    fn read_by_model(&self, model_hash: &str) -> Result<Vec<CalibrationPoint>>;
    fn clear_model(&self, model_hash: &str) -> Result<()>;
    fn model_hashes(&self) -> Result<Vec<String>>;
}
```

**第二，状态机的时间窗口与 `--interval` 参数存在隐式耦合。** 方案中状态转换基于 `Instant::now()` 和固定秒数，这没问题。但若用户在 `--interval 60` 模式下运行，一次采集间隔就是 60 秒。`warning_seconds = 10` 意味着可能连续 1 次采集就触发状态转换，但实际上只过了 60 秒。这会导致状态切换太快，失去平滑意义。建议状态转换需要同时满足"持续时间"和"最少采集次数"两个条件，例如：high 持续 10 秒 **且** 至少采集 3 次。防止用户设了极长的 interval 时，一次异常就切状态。

**第三，GPU 温度/功耗字段的 `Option` 嵌套可能给 Agent 解析带来困扰。** 当前设计 `gpu_temps_c: Vec<Option<f64>>` 和 `power_watts: Option<f64>`。Agent 拿到后需要处理两层 None。建议统一为：如果某 GPU 不支持温度采集，则整个 `temperature_c` 字段不存在（而非数组元素为 null），即在序列化时 `skip_serializing_if = "Option::is_none"`。这样 Agent 的解析逻辑更简单。

**结论**：通过。要求 `CsvStore` 抽象为 trait、状态机增加最少次数条件、GPU 指标序列化优化。


## 👤 产品经理

技术方案的模块划分清晰，但从用户交互和体验一致性角度看，有几个细节需要补上。

**第一，`--calibration-stats` 的输出太"技术化"。** 方案里是冷冰冰的数字。但用户看这个是想知道"我的 Agent 学得怎么样了"。建议输出包含一个简单的进度条或等级描述：例如样本数 3/10 时显示"学习阶段 ███░░░░░░░ 30%"，10/10 时显示"已校准 ✅"。这在终端彩色输出中实现并不复杂，但体验差异巨大。

**第二，温度模块的"最简实现"只采集不预警，这个边界需要向用户明确传达。** 如果 W4 决定裁剪，那么 `--json` 输出中会出现 `thermal` 字段但 `thermal_action` 始终为 `null`。用户会困惑"怎么有温度数据但不告诉我该不该停"。建议在最简实现中，即使不自动给出 `thermal_action`，也要在 `reason` 或 `_note` 中加一句："温度数据仅供参考，自动预警将在后续版本提供。" 消除困惑。

**第三，多 Agent 检测的"全局争抢预警"策略太简单粗暴。** 只要总内存超过 80% 就报 `global_pressure: "high"`，但没有考虑 Agent 的优先级或资源配额。建议在 V0.3 仅输出检测结果，不输出 `global_pressure`。策略协调留到 V0.4 做，避免给出不成熟的建议。否则用户以为秋毫mem 能协调，用了发现不能，反而失望。

**结论**：通过。要求 `--calibration-stats` 增加进度条/等级描述、温度最简实现增加说明文案、V0.3 多 Agent 只检测不预警。


## 👤 安全专家

技术方案对隐私和安全的设计基本到位，但我有几点补充。

**第一，CSV 存储模型名哈希是对的，但哈希截取 8 位有碰撞风险。** 8 位十六进制仅有 4^8 ≈ 40 亿种可能，对于个人使用完全足够，但作为设计原则，建议使用至少 16 位。这不增加存储成本，但能显著降低碰撞概率。同时，哈希加盐（salt）可以防止彩虹表反推，但 salt 存储在本地，意义不大，可以不强制。

**第二，状态机的"首次采集不延迟"是一个安全设计亮点，但缺少"突然恶化"的快速通道。** 当前逻辑从 Normal 到 Critical 需要经历 Warning 持续 15 秒。如果在 `--interval` 间隔内，可用内存突然从 8GB 骤降到 500MB（例如突然启动一个大模型），状态机需要等 10 秒才进入 Warning，再等 15 秒才 Critical。这 25 秒内 Agent 可能已经 OOM 了。建议增加一条紧急规则：当某次采集的 available_mb 低于绝对阈值（如 < 512MB 或 used_percent > 98%）时，**立即**跃迁到 Critical 状态，绕过时间窗口。这是给 Agent 的最后一道保险。

**第三，进程检测读取 `/proc/*/comm` 而不是 `cmdline`，这个设计是正确的。** 但需要确认在 macOS 上使用 `libproc` 的 `proc_name` 或 `proc_pidinfo` 时，同样只读取进程名。macOS 的进程名可以通过 `libproc::proc_name(pid)` 获取，它对应的就是 `comm` 而不是 `cmdline`，安全。建议在代码注释中明确这点。

**结论**：通过。要求模型哈希至少 16 位、状态机增加紧急快速通道、macOS 进程检测确认只读进程名。


## 👤 PMO

工期 8 周的预估基于 P0 模块顺利交付的前提。但有几个进度风险需要提前预警。

**第一，W4 的裁剪评估点非常关键。** 如果届时 P0 模块（校准+状态机+GPU）出现阻塞，需要果断裁剪 P1。建议在 W3 周五做一次预评估，如果校准引擎或 GPU 任一模块进度落后超过 2 天，就提前启动裁剪讨论，而不是等到 W4。

**第二，GPU Apple Silicon 的 Metal API 验证风险最高。** 技术方案提到"W1 最简验证 MTLCopyAllDevices"。建议把这个验证的 DDL 明确为 W1 周五。如果到周五 Metal API 不可用，立即决策：回退 sysctl 估算，还是延期 Metal 支持。不要拖到 W5。

**第三，测试环境缺口。** V0.3 需要 NVIDIA 带 GPU 的机器、AMD ROCm 机器、Apple Silicon 设备各至少一台，用于验证 GPU 采集。建议在 W1 就锁定这些测试资源。如果缺少 AMD ROCm 环境，AMD 功能标记为"社区测试"。

**结论**：通过。要求 W3 周五预评估、Metal API 验证 DDL W1 周五、提前锁定三平台测试环境。

## ✅ 评审结论

**技术负责人最终裁定**：V0.3 技术方案通过，准予进入开发阶段。

**前置条件**：CR-01、CR-02、CR-05、CR-06、CR-07、CR-08、CR-10、CR-11 必须在编码前或 W1 内落地。CR-03、CR-04、CR-09 在对应模块开发时体现。

**修订后的开发顺序**：
- **W1 前半周**：CR-01（CalibrationStore trait）+ CR-10（Metal API 验证）+ CR-11（锁定测试环境）
- **W1 后半周 ~ W2**：CsvStore 实现 + 校准数据采集 + MCP 改造
- **W2 ~ W3**：CR-02（状态机双条件）+ CR-08（紧急快速通道）+ 状态机主体
- **W3 周五**：CR-09（预评估）
- **W3 ~ W4**：校准修正算法 + confidence 升级/降级
- **W4**：决策点：根据预评估决定是否裁剪 P1
- **W4 ~ W5**：GPU NVIDIA + AMD
- **W5 ~ W6**：GPU Apple Silicon + 温度采集（CR-05）
- **W6 ~ W7**：多 Agent 检测（CR-06）+ CLI 参数
- **W7 ~ W8**：集成测试 + 文档 + 发布
