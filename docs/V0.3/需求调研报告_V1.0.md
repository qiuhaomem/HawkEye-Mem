# 秋毫mem V0.3 · 需求调研报告

**文档编号**：HM-PRD-003
**版本**：V1.0
**日期**：2026-05-20
**编写人**：高级产品经理

---

## 一、调研背景与目标

V0.2 解决了"Agent 住进去之前先看房"的问题。但 Agent 住久了，秋毫mem 应该越看越准。当前 V0.2 的估算引擎仍然依赖用户手动配置模型参数，绝大多数用户跳过这一步，导致估算始终停留在 `confidence: "conservative"`，精度有限。

V0.3 的核心命题：**从"保守估计"到"精准自知"——Agent 越住越了解自己，秋毫mem 越用越准。**

同时，V0.2 的 GPU 监控仅实验性支持 NVIDIA，需要扩展为完整的多平台方案。连续监控模式下的单点误报问题也需要解决。

## 秋毫mem V0.3 · 五级需求分解表

---

### 应用端 1：动态校准引擎 —— Agent 越住越了解自己

| 一级 | 二级 | 三级 | 四级 | 五级 |
|------|------|------|------|------|
| **动态校准引擎** | **校准数据采集** | **推理前后内存快照** | Agent 在推理前调用秋毫mem，记录 `before_snapshot` | 推理前采集：`available_mb_before`、`timestamp_before` |
| | | | Agent 在推理后调用秋毫mem，记录 `after_snapshot` | 推理后采集：`available_mb_after`、`timestamp_after`；Agent 需提供 `tokens_processed`（实际处理的 token 数） |
| | | **校准数据点计算** | 计算单次推理的 `bytes_per_token` | `bytes_per_token = (available_before - available_after) * 1024 * 1024 / tokens_processed`；仅当 `tokens_processed > 0` 且内存变化 > 10MB 时记录（过滤噪声） |
| | **校准数据存储** | **本地 CSV 存储** | 追加写入 `~/.config/hawk-eye-mem/calibration.csv` | CSV 格式：`timestamp,model_name,bytes_per_token,tokens_processed`；无 header，每次校准追加一行；人类可读，零依赖 |
| | | **数据量上限** | 最多保留 100 条记录 | 超出时删除最旧的 50 条；防止文件无限增长 |
| | | **多模型隔离** | 按 `model_name` 分组存储 | 不同模型的校准数据独立计算，不互相干扰；未指定模型名时使用 `"unknown"` |
| | **动态修正算法** | **加权平均计算** | 从 CSV 中读取同模型的所有记录 | 最近 10 次记录权重更高（线性衰减：最新记录权重 1.0，第 10 条权重 0.1）；样本数越多，修正幅度越小（避免单次异常值影响） |
| | | **confidence 自动升级** | 样本数 ≥ 10 且标准差 ≤ 20% | `confidence` 自动从 `"conservative"` 升级为 `"calibrated"` |
| | | **confidence 自动降级** | 连续 3 次实测值与估算值偏差 > 30% | `confidence` 降回 `"conservative"`，清空该模型校准数据，重新学习 |
| | | **冷启动回退** | 样本数 < 10 或校准数据不可用 | 回退到 V0.2 保守公式（默认 `bytes_per_token=2048`，`margin=30%`） |
| | **校准数据输出** | **JSON 增加校准元数据** | 在 `agent_guidance` 中增加 `calibration` 字段 | 包含：`samples`（样本数）、`avg_bytes_per_token`（加权均值）、`std_dev`（标准差）、`trend`（"stable"/"improving"/"degrading"） |
| | | **校准建议** | 样本数 < 10 时输出提示 | `suggestion: "Run more inference tasks to improve estimation accuracy. Current samples: 3/10."` |

**约束条件**：
- 动态校准仅在使用 `--model` 参数或配置了 `model.name` 时启用。
- 若未指定模型名，秋毫mem 不记录校准数据，仍使用保守公式。
- CSV 文件损坏时，静默删除并重新创建，不报错。
- 校准数据仅在本地存储，不上传任何服务器。

**与产品经理评审决议的对齐**：校准后 `confidence` 变为 `calibrated`，但 `_note` 字段仍保留温和免责提示——"校准后估算更接近实际，但仍有偏差可能。"

### 应用端 2：连续监控状态机 —— 消除单点误报

| 一级 | 二级 | 三级 | 四级 | 五级 |
|------|------|------|------|------|
| **连续监控状态机** | **状态定义** | **三态模型** | `normal` / `warning` / `critical` | 初始状态为 `normal` |
| | **状态转换** | **normal → warning** | 连续 3 次采集 pressure ≥ `high` | 转换时输出 `action: "monitor"`；连续计数器在每次采集后更新 |
| | | **warning → critical** | 连续 5 次采集 pressure = `critical` | 转换时输出 `action: "abort_safely"` |
| | | **warning → normal** | 连续 3 次采集 pressure ≤ `medium` | 转换时输出 `action: "ok"` |
| | | **critical → warning** | 连续 3 次采集 pressure ≤ `high` | 转换时输出 `action: "reduce_context"` |
| | **状态保持** | **同状态内不重复报警** | 状态未变化时，`action` 保持不变 | 但 `reason` 仍每次更新（反映最新数值） |
| | **首次采集行为** | **冷启动不延迟** | 首次采集时状态为 `normal`，但若 pressure ≥ `high`，立即进入 `warning` | 首次采集不等待连续计数 |
| | **状态重置** | **`--count` 结束后重置** | 新一轮 `--count` 或 `--interval` 启动时，状态机归零 | 避免跨会话的状态污染 |

**约束条件**：
- 状态机仅在 `--interval` 模式（连续监控）下启用。
- 单次调用（无 `--interval`）不使用状态机，直接输出即时判定。
- 状态转换的连续次数阈值可通过配置文件调整（默认 3/5）。

### 应用端 3：GPU 完整支持 —— 三大平台覆盖

| 一级 | 二级 | 三级 | 四级 | 五级 |
|------|------|------|------|------|
| **GPU 完整支持** | **NVIDIA（增强）** | **NVML 直接绑定** | 替代 V0.2 的 nvidia-smi 解析方案 | 使用 `nvml-wrapper` crate，直接调用 NVML API；采集：`name`、`vram_total_mb`、`vram_used_mb`、`temperature_c`、`power_watts`；编译条件：`#[cfg(feature = "gpu")]` |
| | | **nvidia-smi 降级方案** | NVML 不可用时自动降级 | 若 NVML 初始化失败（驱动不兼容），自动降级为 V0.2 的 nvidia-smi 解析方案；降级时 stderr 输出 warning |
| | **AMD（新增）** | **ROCm 支持** | 采集 AMD GPU 显存和温度 | 使用 `rocm-smi` 命令行工具解析；采集：`name`、`vram_total_mb`、`vram_used_mb`、`temperature_c`；`rocm-smi` 不在 PATH 时静默跳过 |
| | **Apple Silicon（新增）** | **Metal API 采集** | 采集统一内存中 GPU 部分的占用 | macOS 上使用 Metal API（通过 `metal-rs` 或 `objc` 绑定）；采集：`name`、`vram_total_mb`（统一内存总量）、`vram_used_mb`（GPU 端当前占用估计）；若 Metal API 不可用，回退到 sysctl 估算 |
| | **多 GPU 支持** | **多卡聚合** | 同时采集多块 GPU | `system.gpu` 为数组，每个 GPU 一个元素；多 GPU 时各 GPU 独立判定 `pressure` |
| | **GPU 温度监控** | **温度采集** | 在 `GpuMetrics` 中增加 `temperature_c` | NVIDIA: NVML 直接获取；AMD: `rocm-smi` 解析；Apple Silicon: 通过 `pmset -g thermlog` 或系统 API 获取 SOC 温度 |
| | **GPU 功耗监控** | **功耗采集** | 在 `GpuMetrics` 中增加 `power_watts` | NVIDIA: NVML 直接获取；AMD/Apple Silicon 暂不支持，字段为 `null` |

**约束条件**：
- 所有 GPU 采集模块均通过 `#[cfg(feature = "gpu")]` 条件编译。
- 未启用 feature 时，`system.gpu` 字段不存在（与 V0.2 行为一致）。
- GPU 采集失败不影响其他资源采集，对应 GPU 元素标记 `error` 字段。

### 应用端 4：温度/功耗监控

| 一级 | 二级 | 三级 | 四级 | 五级 |
|------|------|------|------|------|
| **温度/功耗监控** | **CPU 温度** | **温度采集** | Linux: `/sys/class/thermal/thermal_zone*/temp` | 读取热区温度，取最大值；macOS: `pmset -g thermlog` 或 SMC 估算 |
| | | **温度压力判定** | 三档：`normal`（<80°C）/ `warning`（80-95°C）/ `critical`（>95°C） | 判定逻辑与内存/磁盘/CPU 压力一致 |
| | **GPU 温度** | **温度采集** | 见 GPU 完整支持模块 | 各 GPU 独立上报温度 |
| | **温度墙预警** | **降频预警** | 当 CPU 或 GPU 温度接近降频阈值（通常 95-100°C） | 输出 `thermal_throttle_warning: true`；`agent_guidance` 增加 `thermal_action`：`"reduce_load"` 或 `"pause"` |
| | **功耗监控** | **CPU 功耗** | 暂不支持（需要 RAPL 或类似接口，跨平台一致性差） | V0.3 不做，V0.4 评估 |
| | **thermal 字段输出** | **JSON 新增字段** | `system.thermal` | 包含 `cpu_temp_c`、`gpu_temps_c`（数组）、`throttle_warning` |

**约束条件**：
- 温度采集需要平台特定权限，无权限时对应字段为 `null`，不报错。
- macOS 上 CPU 温度可能需要 SMC 访问，若无法获取则回退到系统报告的 SOC 温度。

### 应用端 5：多 Agent 资源感知

| 一级 | 二级 | 三级 | 四级 | 五级 |
|------|------|------|------|------|
| **多 Agent 感知** | **进程检测** | **已知框架进程名匹配** | 扫描进程树 | 内置列表：`hermes`、`claude-code`、`autogpt`、`langchain`、`open-interpreter`；用户可通过配置文件扩展 |
| | | **Agent 进程统计** | 输出检测到的 Agent 数量 | `agents_detected` 数组，每个元素包含 `name`、`pid`、`memory_rss_mb`、`cpu_percent` |
| | **资源争抢预警** | **全局内存占用估算** | 累加所有检测到的 Agent 进程内存 | 若总占用 > 80% 物理内存，输出 `global_pressure: "high"` |
| | | **建议协调策略** | 多 Agent 场景下的建议 | 简单策略：按内存占用排序，建议内存占用最大的 Agent 先缩上下文；不自动执行，仅输出建议 |
| | **JSON 输出扩展** | **multi_agent 字段** | `system.multi_agent` | 包含 `agents_detected` 数组、`total_agent_memory_mb`、`global_pressure` |

**约束条件**：
- 进程检测依赖平台特定的进程遍历 API（Linux: `/proc`；macOS: `sysctl` 或 `libproc`）。
- 无权限或无法遍历进程时，`multi_agent` 字段为 `null`，不报错。
- 仅检测，不干预；不自动 kill 或 renice 任何进程。

### 八、V0.3 与 V0.2 对比

| 维度 | V0.2 | V0.3 |
|------|------|------|
| 估算精度 | 手动配置或保守默认 | 自动学习，从保守进化到精准 |
| GPU 支持 | NVIDIA 实验性 | NVIDIA + AMD + Apple Silicon 完整支持 |
| 连续监控 | 每次独立判定，可能误报 | 状态机平滑，消除单点尖峰 |

| 温度/功耗 | 无 | CPU/GPU 温度 + 降频预警 |

### 九、版本优先级（MoSCoW）

| 优先级 | 需求 |
|--------|------|
| **Must Have** | 动态校准引擎（采集+存储+修正）、连续监控状态机、GPU NVIDIA 增强+AMD+Apple Silicon、校准数据 CSV 存储 |
| **Should Have** | 温度/功耗监控、多 Agent 基础检测、`--calibration-stats` / `--reset-calibration` / `--gpu-list` CLI |
| **Could Have** | 多 Agent 协调建议、GPU 功耗（仅 NVIDIA） |
| **Won't Have** | 远程采集、环境指纹、守护进程（这些都是 V0.4/V1.0） |

---

**产品经理总结**：V0.3 的核心是"越用越准"。动态校准引擎让秋毫mem 从一次性工具变成持续学习的伙伴——Agent 住得越久，秋毫mem 越了解它。连续监控状态机解决 V0.2 的误报痛点。GPU 完整支持覆盖所有主流平台，不再让 AMD 和 Apple Silicon 用户二等公民。温度和多 Agent 是加分项，但不阻塞核心交付。
