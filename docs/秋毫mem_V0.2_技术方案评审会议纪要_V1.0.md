### 👤 架构师

方案整体架构方向正确，`ResourceCollector` trait 的统一抽象解决了 V0.1 扩展性不足的问题。但有两个设计缺陷必须在编码前修正：

**第一，`CollectorRegistry` 的串行采集在 V0.2 不是问题，但当前设计给未来扩展埋了坑。** 每个 Collector 的 `collect` 方法接收 `&mut ResourceSnapshot`，这意味着并行采集时会产生 borrow 冲突。建议改为 Collector 返回自己的指标，由 Registry 负责组装：

```rust
// 当前设计（不推荐）
fn collect(&self, snapshot: &mut ResourceSnapshot) -> Result<()>;

// 建议改为（可并行友好）
fn collect(&self) -> Result<CollectorOutput>;

enum CollectorOutput {
    Memory(MemoryMetrics),
    Disk(DiskMetrics),
    Cpu(CpuMetrics),
    Gpu(Vec<GpuMetrics>),
}
```

**这样改的好处**：每个 Collector 完全独立，未来可以 spawn 多个线程并行采集，Registry 收到结果后组装成 `ResourceSnapshot`。改动成本很小，但架构收益大。

**第二，`AssessmentEngine` 的降级方案生成逻辑需要更明确的优先级定义。** 方案里写了"降低量化→降低上下文→换小模型"，但这个顺序在特定场景下不最优。比如用户指定了 Q2_K 量化（已经是最低了），第一步降量化就走不通。建议把降级方案的生成逻辑改为决策树：

```
输入：原请求 + 约束列表
  ├─ 约束项 = 内存不足？
  │   ├─ 量化可降？→ 方案A：降量化
  │   ├─ 上下文可降？→ 方案B：降上下文
  │   └─ 有更小模型？→ 方案C：换小模型
  ├─ 约束项 = 磁盘不足？
  │   └─ 方案D：提示清理磁盘空间（无法自动降级）
  └─ 约束项 = GPU显存不足？
      └─ 同内存逻辑，但针对显存
```

每个约束项独立生成降级方案，最后组合输出。

**结论**：通过。要求 Collector 返回独立结果而非修改共享引用、降级方案改为决策树逻辑。


### 👤 产品经理

技术方案的模块划分没问题，但从用户交互角度看，有几个体验细节被忽略了。

**第一，`--can-run` 的失败输出需要区分"彻底不能"和"勉强能跑但差一点"。** 用户看到 `infeasible` 只会沮丧。但如果同时告诉他"差多少"——比如"内存差 344MB，如果你关掉 Chrome 就能跑了"——这个信息是有行动价值的。方案里 `constraints` 数组已经有 `gap_mb` 字段，但缺少**可操作的补救建议**。建议每个 constraint 的 `suggestion` 字段不仅告诉用户降级模型参数，也告诉用户"如果你释放 X MB 内存，就可以部署当前配置"。

**示例**：
```json
{
  "resource": "memory",
  "gap_mb": 344,
  "suggestion": "方案A：降低量化到Q2。方案B：释放至少344MB内存（例如关闭浏览器或IDE）即可用当前配置部署。"
}
```

**第二，`--compare` 的输出如果只是两个 JSON 块并排，用户要自己逐行对比，体验很差。** 建议在人类可读模式下，对比输出用表格，并且**高亮推荐项**——自动标注哪个模型更适合当前环境。

**第三，`--list-models` 的彩色表格设计需要提前定下来，不要等到 W4 才想。** 我建议现在就把输出格式确定：

```
模型名              参数量      推荐量化     最低内存     来源
llama3-8b          8.0B        Q4_K_M       6.0 GB      官方预置
qwen2-7b           7.0B        Q4_K_M       5.5 GB      官方预置
...
```

绿色 = 当前机器可以跑，黄色 = 勉强能跑，红色 = 跑不动。颜色判定基于当前系统内存快照。

**结论**：通过。要求 constraint 的 suggestion 增加环境侧补救建议、`--compare` 高亮推荐、`--list-models` 颜色判定基于实时快照。


### 👤 安全专家

方案中 GPU 采集涉及调用外部二进制和链接专有库，安全风险需要关注。

**第一，`nvidia-smi` 解析方案的风险。** `nvidia-smi` 的输出格式在不同驱动版本间有差异。`--query-gpu` 的 CSV 输出相对稳定，但字段顺序可能在极少数情况下变化。建议在解析时按 header 行匹配字段位置，而非硬编码列索引。同时，`Command::new("nvidia-smi")` 依赖 PATH 环境变量，如果用户机器上 nvidia-smi 不在 PATH 中，应该给出明确提示而非静默失败。

**第二，NVML 绑定的内存安全。** 如果使用 `nvml-wrapper` crate，其内部有 unsafe 代码。V0.1 我们对 unsafe 的管理策略是"集中在 macOS sysctl_val 并加注释"。V0.2 应该延续这个策略：所有 NVML 调用封装在单一模块内，模块顶部显式声明 `#[allow(unsafe_code)]` 并注释原因。

**第三，磁盘路径检测的信息泄露风险。** `DiskCollector` 会检测 `~/.cache/huggingface/` 等路径是否存在。这些信息出现在 JSON 输出中，如果日志或输出被分享到公网，相当于暴露了用户的文件系统布局。建议在 JSON 输出中对路径做脱敏处理：`/home/username/.cache/models` → `~/.cache/models`。这不是安全漏洞，但这是一个好的隐私习惯。

**结论**：通过。要求 nvidia-smi 解析按 header 匹配字段、NVML unsafe 集中封装、磁盘路径在输出中脱敏。


### 👤 PMO

技术方案的风险点识别和应对措施基本到位。工期 6 周的预估合理。但有三个执行层面的事需要明确。

**第一，W1 的 trait 重构是 V0.2 最关键的工程基础。** 如果重构引入回归 bug，后续所有功能都建立在有问题的地基上。建议 W1 重构完成后，跑三轮测试：V0.1 原有单元测试 + V0.1 原有集成测试 + 新增的 trait 接口测试。三轮全部通过才进入 W2。

**第二，GPU 验证的"1 天止损"需要明确定义。** "1 天"是 8 个工时还是 24 小时？如果是工作日，跨周末怎么算？建议定义为"连续 8 个工时投入后仍未通过"，超时自动触发切换。止损后原 NVML 方案不会丢弃，作为分支保留，等社区贡献或后续版本再启用。

**第三，V0.2 的发布 checklist 需要在 W5 提前准备。** 包括：
- V0.1 → V0.2 的升级指南（配置文件兼容性说明）
- 新增 CLI 参数的 help 文档
- README 更新（V0.2 功能描述 + 部署评估示例）
- DISCLAIMER 更新（部署评估免责条款）
- 4 个 target 的编译验证（x86_64-linux-musl, aarch64-linux-musl, x86_64-macos, aarch64-macos）

**结论**：通过。要求 W1 重构后三轮测试、GPU 止损定义为 8 个工时、W5 提前准备发布 checklist。
