# 秋毫mem V0.4 · 需求调研报告

**文档编号**：HM-PRD-004
**版本**：V1.0
**日期**：2026-05-21
**编写人**：高级产品经理

## 一、调研背景与目标

V0.3 实现了动态校准，让 Agent 在一台机器上越住越了解自己。但现实是：开发者的 Agent 往往先在笔记本上调试，然后部署到云服务器或企业内部服务器上。环境一变，Agent 又成了瞎子。

V0.4 的核心命题是：**Agent 搬家也不怕——当它从一台机器迁移到另一台，秋毫mem 能自动感知环境变化，并给出新的部署建议和运行策略。同时，支持远程采集多台机器的资源状态，为多 Agent 协调提供基础。**

## 二、用户调研

| 来源 | 痛点 | 频率 |
|------|------|------|
| 内部试用反馈 | Agent 从 M1 Mac 部署到 Linux 服务器后，原来的校准数据没用了，一切从头来 | 高 |
| r/LocalLLaMA 社区 | 开发者想同时监控多台跑模型的机器，不想每台都 SSH 上去看 | 中高 |
| 企业用户意向 | 需要历史趋势分析，判断是否需要扩容 | 中 |
| Docker 用户 | 容器内运行的 Agent 感知不到宿主机的真实资源限制 | 高 |
| 多 Agent 场景 | 同一台机器上跑多个 Agent，需要协调资源，而不是各抢各的 | 中高 |

## 四、V0.4 核心功能定义

| 模块 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| **环境指纹** | 环境快照与变更检测 | 首次运行生成环境指纹，后续检测重大变化（内存、CPU、GPU、磁盘），主动通知 Agent | P0 |
| **迁移建议** | 环境变化后自动评估 | 检测到环境升级/降级，自动运行 `--can-run` 逻辑，给出新环境的部署建议 | P0 |
| **远程采集** | 多机资源监控 | 通过 HTTP/gRPC 采集远程机器指标，支持多机聚合视图 | P0（企业版试点） |
| **历史趋势** | 本地存储与趋势分析 | 存储 7 天历史数据，支持趋势预测（预计何时达临界） | P1 |
| **Docker/K8s 适配** | 容器资源感知 | 检测 cgroup 限制，尊重容器内存/CPU 上限 | P1 |
| **多 Agent 协调** | 资源分配建议 | 同机多 Agent 时，给出全局降级策略（Agent A 缩上下文，Agent B 暂停） | P1 |
| **告警输出** | 可消费的告警流 | `--alert` 模式，压力达 critical 时输出带时间戳的告警行，配合外部告警系统 | P2 |
