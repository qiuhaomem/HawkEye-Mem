**评审对象**：V0.4 技术方案设计书（HM-TD-004）
**会议主持**：技术负责人
**参与方**：架构师、产品经理、安全专家、PMO
**创始人列席**
**结论性质**：通过，含 7 条整改要求


## 👤 架构师

方案整体架构演进方向正确，但 V0.4 引入了网络服务、新的存储文件和容器适配层，复杂度显著增加，有几个设计点如果不提前处理，V1.0 会变成技术债的重灾区。

**第一，`--serve` 的 HTTP 服务使用了 `tiny_http`，这是一个单线程阻塞的 HTTP 库。** V0.4 的远程采集场景可能同时有多个 Agent 查询同一个秋毫mem 实例。单线程阻塞意味着一个慢客户端会拖住所有请求，这在多 Agent 环境下是致命的。建议切换到 `actix-web` 或 `axum`，或者至少使用线程池。但考虑到我们一直坚持"不引入 tokio"的原则，可以用标准库的 `std::net::TcpListener` + `thread::spawn` 实现一个极简的多线程 HTTP 服务，不引入异步运行时。如果评审认为引入异步运行时可以接受，那建议一步到位用 `axum`。

**第二，环境指纹的变更阈值"模型所需内存的 50%"过于依赖一个没有明确定义的参数。** 用户可能跑 3B 模型，也可能跑 70B 模型。如果用户一直在跑 3B 模型（需要 ~3GB），内存从 16GB 升级到 20GB（增 4GB），超过了 3GB 的 50%（1.5GB），触发迁移建议。但这个建议其实没必要——3GB 模型在 16GB 和 20GB 上都能跑，Agent 不会从 4096 上下文升级到 8192 上下文。建议将阈值改为更简单直接的规则：**绝对值变化超过 4GB 或 20% 总内存（取两者中较大者）时触发**。这样 16GB→20GB（差 4GB，占比 25%，取较大值 4GB）刚好触发；16GB→18GB（差 2GB，占比 12.5%，取较大值 3.2GB）不触发。

**第三，`history.jsonl` 和 `calibration.csv` 虽然都实现了 `CalibrationStore` trait，但两者的数据模型完全不同。** 当前设计中 `CalibrationStore` trait 的 `append` 方法签名是 `(CalibrationPoint, model_hash)`，但历史数据点没有 `model_hash` 概念。建议在 V0.4 中将 trait 进一步抽象为 `TimeSeriesStore`，让校准存储和历史存储都实现这个更通用的接口。这样 V1.0 再引入新的存储类型时，架构不需要再改。

**结论**：通过。要求 HTTP 服务支持多线程/异步、环境指纹阈值改为更简单的规则、`CalibrationStore` trait 进一步抽象。


## 👤 产品经理

技术方案的功能实现路径清晰，但有几个产品体验上的细节需要明确。

**第一，环境指纹变更时输出的 `new_recommendation` 文案不能太机械。** 当前方案中只是一个固定格式的字符串。但用户在凌晨三点看到 Agent 发来一条"环境已变更"通知时，最需要的是一个具体的、可操作的建议。建议让 `new_recommendation` 根据变更的方向和幅度动态生成：如果是内存大幅升级（如 16GB→64GB），告诉 Agent"你以前只能跑 8B，现在可以试试 70B"；如果是小幅降级（如 16GB→12GB），告诉 Agent"上下文从 8K 降到 6K 比较保险"；如果是 GPU 从无到有，告诉 Agent"你现在有 GPU 了，可以切换推理后端"。

**第二，`--trend` 输出中的"预计 X 天后到达临界"是一个面向未来的预测，但用户需要一个"我现在就该担心吗"的判断。** 如果预测 14 天后到达临界，用户可能觉得还早。但如果趋势斜率在加速，14 天后可能变成 7 天。建议在趋势输出中增加一个 `urgency` 字段（`low` / `medium` / `high`），综合趋势方向和斜率判断。例如：临界距离 > 30 天 → `low`；7-30 天 → `medium`；< 7 天 → `high`；已临界 → `critical`。

**第三，`--alert` 模式只输出"一行 critical 告警"但缺少上下文。** 用户或 Agent 拿到告警后还需要再跑一次 `--json` 才知道具体什么资源不够了。建议 `--alert` 模式输出一个最小化的 JSON 单行，包含 `pressure`、`available_mb`、`action` 三个字段。这样告警接收方不需要二次查询就能做出判断。

**结论**：通过。要求 `new_recommendation` 动态生成、`--trend` 增加 `urgency` 字段、`--alert` 输出最小化 JSON。


## 👤 安全专家

V0.4 首次引入网络服务，安全设计基本到位，但有几个细节需要加强。

**第一，`--serve` 的 API Key 认证目前在代码中是基于字符串比较，没有防时序攻击。** 如果攻击者可以尝试不同 API Key 并测量响应时间，可以逐位推断出密钥。建议使用恒定时间比较函数（如 `ring::constant_time::verify_slices_are_equal`）。这不是高优先级问题（因为服务绑定在 localhost），但作为安全最佳实践，应该在代码中实现。

**第二，远程采集的 `/metrics` 端点目前没有速率限制。** 如果多个 Agent 高频查询（比如每个推理步骤前都查一次），服务端可能被压垮。建议在 HTTP 服务中加入简单的速率限制：每个 IP 每秒最多 10 次请求，超过时返回 429 并记录 warning。这在多 Agent 环境下既能保护服务端，也反向约束 Agent 不要过度查询。

**第三，环境指纹存储 `environment.json` 包含机器配置的快照，但没有完整性保护。** 如果用户手动修改了这个文件（比如把内存从 16GB 改成 64GB 假装自己升了个级），秋毫mem 会基于伪造数据给出错误的建议。虽然本地文件的篡改风险较低，但建议在存储指纹时附带一个 HMAC 签名，读取时验签。签名密钥可以是机器 ID 的哈希，这样同机文件不会被别机冒用。

**第四，历史数据 `history.jsonl` 如果被分享到公网或粘贴到 Issue 里，可能暴露用户的工作负载模式。** 建议在 `--trend` 输出中不包含原始数据点，只输出聚合后的趋势指标。原始数据仅保留在本地文件。

**结论**：通过。要求 API Key 比较防时序攻击、`/metrics` 增加速率限制、环境指纹增加完整性签名、`--trend` 不输出原始数据点。


## 👤 PMO

工期 10 周的预估基于所有模块顺利推进，但 V0.4 有多项外部依赖和测试资源需求，需要提前锁定。

**第一，远程采集的多机测试需要至少 3 台机器（2 个服务端 + 1 个客户端）同时运行。** 如果团队没有这么多闲置机器，可以用本地虚拟机或 Docker 容器模拟。建议在 W2 远程采集开发启动时，同步搭建好 3 个 Docker 容器的测试环境，每个容器运行秋毫mem `--serve`。

**第二，容器适配层的测试需要覆盖 Docker（多种 cgroup v1/v2 配置）和 K8s（需要 minikube 或 kind 集群）。** 建议在 W3 搭建 minikube 环境用于 K8s 测试。如果搭建困难，至少覆盖 Docker 场景，K8s 标记为"社区测试"。

**第三，V0.4 的 P1 功能（历史趋势、多 Agent 全局视图）在 W5-W6 开发时，如果进度落后超过 3 天，立即启动裁剪。** P1 最简实现的标准是：历史趋势只输出方向（不输出预测、不输出 urgency）、多 Agent 只输出进程列表（不输出资源占用详情）。这样可以确保核心 P0 功能不延期。

**第四，商业化的内部试点（联系 3-5 个种子企业用户）应该在 V0.4 发布后 2 周内启动。** 建议在 V0.4 的 Release Notes 中不提及远程采集为企业版功能，但在 GitHub Discussions 中私下联系活跃的企业用户，邀请他们试用并反馈。

**结论**：通过。要求提前搭建 Docker 多机测试环境、K8s 测试环境降级为可选、W5 评估 P1 裁剪、发布后 2 周内启动企业用户试点。

## ✅ 评审结论

**技术负责人最终裁定**：V0.4 技术方案通过，准予进入开发阶段。

**前置条件**：CR-01、CR-02、CR-04、CR-06、CR-08、CR-10、CR-11 必须在 W1-W2 内落地。CR-03、CR-05、CR-07、CR-09 在对应模块开发时体现。

**修订后的开发顺序**：
- **W1-W2**：CR-01（多线程 HTTP）+ CR-02（指纹阈值）+ CR-04（动态文案）+ 环境指纹引擎主体
- **W2-W3**：CR-08（速率限制）+ 远程采集 HTTP 服务端 + 客户端
- **W3**：CR-11（Docker 测试环境）
- **W3-W4**：容器适配层
- **W4-W5**：趋势分析引擎（CR-05 urgency 字段、CR-10 不输出原始数据）
- **W5**：CR-11（P1 裁剪评估点）
- **W5-W6**：多 Agent 全局视图 + CLI 新增参数（CR-06 `--alert` 最小化 JSON）
- **W6-W8**：集成测试 + 跨平台 + 多机
- **W8-W9**：文档 + 数据说明文档
- **W9-W10**：Buffer + 发布
