# 秋毫mem V0.4 · 需求清单 多方评审会议纪要

**评审对象**：V0.4 需求调研报告 + 五级需求分解表
**会议主持**：高级产品经理
**参与方**：技术负责人、安全专家、市场运营、法务、PMO
**创始人列席**
**结论性质**：通过，含 9 条整改要求

## 👤 技术负责人

需求方向没问题，但实现上有几个点必须现在说清楚，否则开发阶段全是坑。

**第一，远程采集的 HTTP 服务 `--serve` 需要明确传输协议和数据格式。** 需求里写了"HTTP 服务，提供 `/metrics` 端点，返回当前资源快照 JSON"。但秋毫mem 的 JSON 输出现在有三层（`system`、`agent_guidance`、可选的 `deployment_assessment`）。远程采集应该只返回 `system` 层，不需要 `agent_guidance`（远程机器的建议对本地 Agent 没意义，还会浪费带宽）。建议在需求中明确：`/metrics` 端点只返回 `system` 快照，不带建议信息。`/full` 端点可选返回完整输出。

**第二，环境指纹的"核心资源变化 > 20%"这个阈值太粗糙。** 如果用户从 16GB 升级到 20GB（变化 25%），触发迁移建议。但实际上 4GB 的增量对部署建议的影响很小，频繁触发反而让 Agent 困惑。建议改为：只在资源变化量超过模型所需最小资源的 50% 时才触发建议。例如用户一直跑 8B 模型（需要约 6GB），只有当内存变化超过 3GB 时才重新评估。或者增加一个"静默模式"，用户可以手动关闭迁移建议。

**第三，Docker 容器内读取 cgroup 内存限制有一个常见陷阱。** `memory.limit_in_bytes` 在没有设置限制时返回一个巨大的值（如 9.22e18），不能直接当作物理内存。需要判断：若 limit 大于或等于物理内存，使用物理内存；否则使用 limit。K8s 环境下同样适用。另外，`cpu.cfs_quota_us` 和 `cpu.cfs_period_us` 可能为 -1（无限制），需要处理。

**第四，历史数据存储 `history.jsonl` 与校准 CSV 的关系需要厘清。** 两者都存储时间序列数据，但一个是推理校准用，一个是趋势分析用。能否统一为一个存储？或者至少共享同一个数据管理逻辑？避免两个文件各自增长、各自清理，增加维护成本。建议历史数据复用 `CalibrationStore` trait（V0.3 已抽象），只是存储内容不同。

**结论**：通过。要求明确远程采集端点数据范围、环境指纹阈值改为模型相关、cgroup 无限值处理、历史数据复用存储抽象。

## 👤 安全专家

V0.4 首次引入网络服务，安全风险显著增加，需要重点关注。

**第一，`--serve` 模式默认绑定 localhost 是对的，但需求文档里写"不支持公网暴露"不够强硬。** 应该明确：`--serve` 启动时主动检查绑定地址，如果是 `0.0.0.0` 或公网 IP，必须输出醒目的红色警告并等待 5 秒后强制退出。这不是限制用户自由，是防止新手误操作把秋毫mem 暴露到公网上被扫描。

**第二，远程采集的 API Key 认证目前是明文传输。** 需求里没有明确传输方式。如果走 HTTP 明文，API Key 在局域网内也可能被嗅探。建议在需求中明确：`--remote` 优先尝试 HTTPS；如果用户提供的是 `http://` URL，必须输出警告但允许继续。同时建议支持从环境变量读取 API Key，避免命令行泄露。

**第三，历史数据 `history.jsonl` 和 `environment.json` 可能包含主机名。** 在容器环境中，主机名可能泄露内部命名规则或项目代号。建议在存储前对敏感字段（主机名、路径）做脱敏，和 V0.3 的模型名哈希保持一致策略。

**第四，远程采集的聚合视图如果包含多台机器的详细数据，意味着一次请求暴露了整个集群的资源状态。** 如果攻击者拿到 API Key，就能获取整个集群的负载信息。建议在需求中增加"最小权限原则"：远程采集的 `/metrics` 端点不应返回所有字段，只返回聚合任务所需的最小指标集（如内存可用量、CPU 负载、磁盘剩余）。

**结论**：通过。要求 `--serve` 公网绑定硬拦截、API Key 传输安全提示、历史数据脱敏主机名、远程端点最小权限。

## 👤 市场运营

V0.4 的"搬家也不怕"叙事很强，但有两个传播风险需要提前处理。

**第一，"商业化试点"这个词在 V0.4 发布时不能在公开渠道出现。** 远程采集如果标记为"企业版试点"，社区会立刻解读为"秋毫mem 开始收费了"，这对开源社区的信任是致命打击。建议在 V0.4 的公开文档中完全不用"企业版"、"付费"、"试点"这些词。远程采集作为普通功能发布，标注"实验性"。等 V1.0 社区成熟后再讨论商业化。

**第二，多 Agent 协调是一个容易被过度解读的功能。** 如果我们在推广中说"秋毫mem 可以协调多 Agent 资源分配"，用户会期待它自动做负载均衡、自动调度。但实际上我们只是输出建议，不做任何自动执行。这个差距会引发负面反馈。建议在对外文案中只说"全局资源视图"，不提"协调"，直到 V1.0 真正实现自动协调。

**结论**：通过。要求 V0.4 公开文档不提商业化、多 Agent 对外只说"全局视图"不提"协调"。

## 👤 法务

远程采集和数据存储带来的法律风险需要逐条处理。

**第一，远程采集涉及"传输"数据，即使是在内网，也需要在 DISCLAIMER 中增加相关声明。** 包括：远程采集的数据内容、传输方式、接收方如何处理这些数据。如果用户用秋毫mem 监控别人的机器（比如运维监控整个集群），用户有责任确保已获得相关授权。这些都需要在免责声明中明确。

**第二，历史趋势数据存储 7 天，这个保留期限本身合规。** 但需要明确：用户如何删除历史数据。需求里没有提到删除方式。建议增加 `--clear-history` 命令，并在文档中告知数据存储位置和手动删除方式。

**第三，环境指纹 `environment.json` 和 `history.jsonl` 的存储位置和隐私影响需要在数据说明文档（Privacy.md 或类似）中披露。** 虽然不上传，但本地存储的信息类型、用途、保留期限、删除方式都应告知用户。这是 GDPR 和《个人信息保护法》的基本要求，即使数据完全本地化。

**结论**：通过。要求 DISCLAIMER 增加远程采集责任条款、增加 `--clear-history` 命令、增加本地数据存储说明文档。

## 👤 PMO

需求数量比 V0.3 多出约 30%，工期需要重新评估，功能裁剪策略需要提前明确。

**第一，V0.4 如果全部实现需求中的所有模块，工期至少 10-12 周。** 但路线图原定 8-10 周。建议在需求评审阶段就做一次优先级裁剪：P0（环境指纹+远程采集+容器适配）必须交付；P1（历史趋势+多 Agent 协调）如果时间紧张，只做最简实现（趋势只输出方向不预测、多 Agent 只输出列表不协调）；P2（告警模式、预言机）延后到 V1.0。

**第二，远程采集的商业化试点被市场运营否决了对外宣传，但我们仍然需要在内部验证这个功能的企业需求。** 建议在 V0.4 发布后，私下联系 3-5 个种子企业用户试用远程采集功能，收集反馈。如果反馈积极，V1.0 的商业化方向就确定了；如果反馈冷淡，V1.0 的 Web 仪表盘和商业化需要重新评估。

**第三，环境指纹和迁移建议功能的测试需要多台不同配置的机器。** 至少在 2 台 Linux 和 2 台 macOS 上验证指纹生成和变更检测。建议在 W1 锁定测试资源。

**结论**：通过。要求 V0.4 做优先级裁剪（P0 必达、P1 最简、P2 延后）、内部试点远程采集而非公开宣传、提前锁定多机测试环境。

| 优先级 | 内容 |
