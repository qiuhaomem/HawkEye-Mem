# 秋毫mem V0.4 · 测试用例文档 多方评审会议纪要

**评审对象**：V0.4 测试用例文档 V1.0（共 155 个用例）
**会议主持**：测试经理
**参与方**：产品经理、技术负责人、安全专家、PMO
**创始人列席**
**结论性质**：通过，含 7 条整改要求


## 👤 产品经理

覆盖面比 V0.3 又上了一个台阶。但有几个产品体验的验收标准需要补上。

**第一，UT-ENV-020 到 UT-ENV-024 验证了迁移建议的触发逻辑，但没有验证文案本身的"可操作性"。** 技术方案评审时我们要求 `new_recommendation` 动态生成具体建议（CR-04），但测试用例只写了"含'can now run larger models'"。这句话对不同场景可能太笼统。建议每个场景的测试用例都明确预期文案中的具体关键词：比如内存 16GB→64GB 时含"70B"，GPU 从无到有时含"switch to GPU inference"，内存降级时含"reduce context to X tokens"。

**第二，UT-TRD-017 到 UT-TRD-020 的 `urgency` 等级判定，缺少一个关键场景：趋势斜率为零但当前值已经接近临界。** 比如用户内存已经用了 88%，虽然不再增长，但一直维持在 88%。按当前逻辑 direction=stable，但 urgency 应该至少是 medium，因为随时可能被一个突发负载推入 critical。建议增加一个测试用例：当前值接近临界（used > 80%）且趋势稳定时，urgency 不应为 low。

**第三，IT-CLI-047 验证了 `--alert` 输出最小化 JSON，但没验证这个 JSON 能被管道正确消费。** 建议增加一个端到端用例：`hawk-eye-mem --alert | while read line; do ... done`，验证每行都是合法 JSON 且下游能解析。

**结论**：通过。要求迁移建议文案含具体操作关键词、增加近临界稳定趋势的 urgency 判定、增加 `--alert` 管道消费验证。


## 👤 技术负责人

测试用例对功能路径覆盖得很好，但边界条件和异常路径有几个缺口。

**第一，远程采集的并发测试 UT-REM-030 只验证了"3 个客户端同时请求不阻塞"，但没有验证极端情况：如果 HTTP 服务端在采集系统指标时发生 panic，会不会把整个服务带崩？** 当前设计是每个请求在线程中处理，但如果 `CollectorRegistry::collect_all()` 在某个线程中 panic，只影响该请求还是整个进程？建议增加一个测试用例：注入一个会 panic 的 Mock Collector，验证其他正常请求不受影响。

**第二，容器适配测试 UT-CTN-003 验证了 cgroup 无限制时回退物理内存，但没有覆盖 cgroup 文件存在但值为负数（如 memory.limit_in_bytes = -1）的场景。** cgroup v2 中 `memory.max` 的无限值写法是 `max` 而不是数字，当前解析逻辑可能在这个场景下出错。建议增加 cgroup 文件内容为 `max` 和 `-1` 的测试用例。

**第三，环境指纹的变更检测 UT-ENV-010 到 UT-ENV-018 测试了单维度变化，但缺少多维度同时变化的场景。** 比如用户同时升级了内存和加装了 GPU，或者换了整台机器（所有维度全变）。建议增加一个多维度变更的综合测试用例。

**第四，`history.jsonl` 的数据清理测试 UT-TRD-003 只验证了"7 天前数据被清理"，但没有验证清理操作本身会不会在 IO 慢的机器上阻塞主流程。** 在 2GB 云服务器上，如果 history.jsonl 积累了几千行，清理操作可能耗时超过 1 秒。建议增加 IO 性能保护：清理操作超时 500ms 则放弃本次清理，不阻塞采集。

**结论**：通过。要求增加 HTTP 服务 panic 隔离测试、cgroup 无限值边界测试、多维度变更综合测试、历史数据清理超时保护。


## 👤 安全专家

网络安全专项测试覆盖了主要风险点，但有几个安全细节需要补上。

**第一，SEC-001 验证了 `--serve --bind 0.0.0.0` 被拦截，但没有验证拦截后进程是否正确退出。** 需要确认退出码为非零，且退出前没有短暂监听在公网端口。建议增加时序验证：启动后 100ms 内端口未被占用。

**第二，`/metrics` 端点的最小权限验证 SEC-003 只检查了字段不含主机名和路径，但没有验证不含进程列表或环境变量。** 建议在测试用例中显式列出"禁止返回的字段"：`hostname`、`process_list`、`env_vars`、`cmdline`、`file_paths`。

**第三，历史数据 `history.jsonl` 的脱敏没有覆盖。** 当前设计只在存储时对主机名做了哈希，但 `environment.json` 中的 `hostname` 是哈希后的，而 `history.jsonl` 本身不存储主机名——但如果用户在多台机器上运行秋毫mem，history.jsonl 中可能存了 `container_runtime` 等可定位信息。建议在安全测试中增加一条：history.jsonl 的每一行可独立脱敏，不含任何可关联到具体机器的标识符。

**结论**：通过。要求验证 `--serve` 拦截后端口未短暂暴露、`/metrics` 禁止字段清单显式验证、history.jsonl 脱敏验证。


## 👤 PMO

测试资源方面有一个缺口和两个时间冲突。

**第一，K8s 测试环境需求中没有明确如果 minikube 环境不可用时的替代方案。** 技术方案评审时已经将 K8s 标记为"可选"，但测试用例中仍然有 UT-CTN-010 到 UT-CTN-013 四个 K8s 用例。建议明确：如果 W3 时 minikube 未成功搭建，这 4 个用例降级为手动测试，在 W8 之前找一台有 K8s 环境的机器完成。

**第二，多机协同专项测试 MA-MULTI-001 需要 3 个 Docker 容器同时运行秋毫mem `--serve`。** 这个测试环境在 W2 搭建，但测试执行在 W7，中间隔了 5 周。需要确保测试环境在此期间不被回收或变更。建议在 W2 搭建完成后，将 Docker Compose 配置文件提交到仓库的 `tests/` 目录下，可随时复现。

**第三，测试用例总数 155 个，其中安全测试 5 个在 W8 执行。但 W8 同时要完成覆盖率检查和补充测试，时间可能冲突。** 建议将安全测试提前到 W7，和多机协同专项测试并行执行。

**第四，`--serve` 多线程测试 UT-REM-030/031 需要并发测试工具（如 `wrk` 或 `ab`），测试环境需要提前安装。**

**结论**：通过。要求 K8s 用例降级预案、Docker Compose 环境提交仓库、安全测试提前到 W7、提前安装并发测试工具。
