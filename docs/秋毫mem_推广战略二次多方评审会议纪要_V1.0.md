# 秋毫mem · 推广战略二次多方评审会议纪要

**评审对象**：推广战略调整方案（V0.2低调蓄力 → V1.0奇点引爆）
**会议主持**：市场运营总监
**参与方**：产品经理、技术负责人、PMO、法务、财务
**创始人列席**
**结论性质**：通过，含 9 条执行要求


### 👤 产品经理

战略方向我完全认同。但有一个关键问题：**V0.2低调发布后，我们怎么确保"种子用户反馈"这件事真的发生？**

如果只是把V0.2放在GitHub上等自然增长，反馈可能来得太慢，或者来得太零散。我建议在V0.2的README底部加一个明确的引导："V0.2是一个早期版本，我们正在收集反馈。如果你试用了`--can-run`，请在这个GitHub Issue下告诉我们：检测准不准？有没有帮到你？"这个Issue就是我们V0.3需求的直接输入源。

另外，关于"在Ollama/llama.cpp的Issue中帮助用户时顺带推荐秋毫mem"——这个动作需要把握分寸。不能在别人的Issue里过度推广自己的项目，否则会被认为是蹭流量。建议只在用户明确描述"下载后OOM"、"不知道能不能跑"这些问题时，用"我之前也遇到过，后来用秋毫mem提前检测了"的口吻提及，最后附上命令示例。

**结论**：通过。要求V0.2 README嵌入用户反馈引导Issue，社区回复严格把握"帮助优先、提及从轻"的分寸。


### 👤 技术负责人

战略调整后，技术侧的节奏需要重新对齐。

**第一，V0.2的低调发布不等于匆忙发布。** 即使不投Reddit和HN，代码质量、测试覆盖、文档完整性不能打折扣。因为V0.2的用户是V1.0的种子，他们对秋毫mem的第一印象会影响后续口碑。不能因为"低调"就放松质量门禁。

**第二，V0.3/0.4的开发周期可能因为竞品动态而需要加速。** 如果V0.2发布后，Ollama或llama.cpp在后续版本中加入了类似"部署前检测"的功能，我们的V0.3/0.4就不能按原计划悠闲推进。需要提前设定一个触发条件：如果竞品在V1.0发布前推出了环境检测功能，我们的V0.3/0.4开发周期自动缩短30%，通过砍掉P2功能来保证V1.0的按时抵达。

**第三，V1.0的"品类定义"需要技术架构的支撑。** 如果只是喊"Agent环境感知层"的口号，但架构上不支持扩展，竞品一个PR就能追平。V1.0的守护进程、Collector插件体系、框架集成——这三个技术基础是护城河的钢筋水泥。需要在V0.3/0.4开发时就为V1.0的架构预留足够的扩展点。

**结论**：通过。要求V0.2低调发布但质量门禁不变、设定竞品触发加速条件、V1.0架构扩展点在V0.3开始预留。


### 👤 PMO

战略调整对项目时间线的影响需要明确。

**原计划**：V0.2（6周）→ V0.3/0.4（8-10周）→ V1.0（8周）→ 总周期约24周
**调整后**：V0.2仍然6周，但V0.3/0.4和V1.0之间增加了一个"市场判断节点"

这个节点设在V0.3/0.4开发完成后。届时需要评估：
- V0.2的用户反馈是否验证了需求方向？
- 是否有竞品已经推出类似功能？
- V1.0的技术基础（守护进程、插件体系）是否已经稳定？

如果三个条件全部满足，V1.0按原计划启动。如果竞品已经抢先，V1.0的开发周期压缩到6周，砍掉Web仪表盘（商业化功能），只保留守护进程+框架集成+预言机这三个核心护城河模块。

另外，V0.2的"低调发布"需要有一个明确的发布时间窗口。建议在V0.2测试完成后，选一个周二下午发布（避开周末，方便周中处理突发问题），不投渠道，只发GitHub Release + 一篇技术博客。

**结论**：通过。要求V0.3/0.4结束后增加市场判断节点、V1.0根据判断结果决定是否压缩、V0.2发布时间定在周二下午。


### 👤 法务

战略调整涉及一个之前没讨论过的法律风险：**"品类定义"的竞争法边界**。

"Agent环境感知层"这个品类名称，如果我们用在推广中，本身不构成法律问题。但如果我们将来在社区中公开说"XX竞品抄袭了我们的Agent环境感知层概念"，这就可能触发不正当竞争的风险。建议在V1.0的推广文案中，只强调"我们定义了Agent环境感知层"，不点名任何竞品，不用"抄袭"、"仿冒"等指控性语言。

另外，V0.2低调发布期间的社区回复需要特别注意。在Ollama/llama.cpp的Issue中推荐秋毫mem，如果被对方维护者认为是在他们的社区里拉用户，可能会引发争端。建议只回复纯技术性的建议，不带任何"我们的工具更好"的比较性描述，并明确标注秋毫mem是开源项目。

**结论**：通过。要求V1.0推广不点名竞品、社区回复不带比较性描述、标注开源身份。


### 👤 财务

战略调整后，推广预算的使用节奏完全变了。

**原计划**：V0.2发布时投入大量精力做渠道推广、可能涉及UP主合作费用。
**调整后**：V0.2几乎零推广预算，仅维持GitHub Release和一篇技术博客的人力成本。所有推广预算集中到V1.0。

这个调整在财务上是更健康的——把资源集中在护城河最深的版本，ROI更高。但有一个风险：如果V1.0的引爆效果不达预期，我们就没有第二次机会了。建议在V1.0发布前，做一次小范围的"推广预演"——找5个秋毫mem的种子用户，让他们提前试用V1.0，收集他们的第一反应。如果他们不觉得"这东西我必须用"，我们还有时间调整V1.0的叙事。

**结论**：通过。要求V1.0发布前做小范围用户预演、推广预算集中到V1.0。

### ✅ 评审结论

**市场运营总监最终裁定**：推广战略调整通过，准予执行。

**核心原则**：
1. **V0.2低调不等于低质**：发布质量不变，社区回复保持谦逊和专业。
2. **V1.0引爆需要验证前置**：在奇点引爆前，通过种子用户预演确认叙事的杀伤力。
3. **竞品不恋战但准备加速**：如果竞品跟进，不纠缠功能对比，而是加快V1.0抵达时间。
4. **品类定义是护城河的核心**：先定义品类，再占领品类。V0.2埋种子，V1.0收果子。
