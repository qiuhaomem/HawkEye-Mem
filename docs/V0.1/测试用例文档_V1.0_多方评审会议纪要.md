## 秋毫mem · 测试用例文档 V1.0 多方评审会议纪要

**评审对象**：测试用例文档 V1.0
**主持人**：高级产品经理
**参与方**：AI 系统编程专家、Hermes/Agent 生态专家、法务、技术负责人
**结论性质**：部分通过，需增补 8 条用例后方可执行


### 👤 高级产品经理评审意见

**核心观点**：测试用例覆盖了功能正确性，但缺少对“Agent 决策质量”的验证。我们的用户是 Agent，测试必须站在 Agent 的视角。

**发现的具体问题**：

1.  **`estimated_safe_context_window` 的边界测试不够**：UT-EE-001 验证了正常内存（16GB）的估算，UT-EE-003 验证了极低内存（512MB）。但**中间水位**（4GB、8GB）恰好是触发 `reduce_context` 和 `monitor` 的临界区域，这些边界值没有覆盖。Agent 在不同压力水位之间的过渡是否平滑，全靠运气。

2.  **JSON 输出缺少“Agent 可解析性”验证**：IT-JSON 系列验证了字段类型和精度，但没有验证 Agent 真正依赖的字段是否稳定存在。例如，Agent 可能直接读取 `agent_guidance.action`，如果某个版本不小心改了字段名，Agent 就会静默失败。需要增加 Schema 一致性测试。

3.  **`suggestion` 字段的生命周期测试缺失**：产品评审会定了规则：`suggestion` 仅在 `confidence == "conservative"` 时存在。UT-GG-005/006 覆盖了，但**没有测试用户配置后 `suggestion` 从有到无的过渡**——即首次运行未配置时 Agent 收到 `suggestion`，配置后下次调用不再收到。这恰好是引导用户配置模型参数的核心体验闭环。

**增补要求**：
-   **UT-EE-005**：边界值测试（4GB、8GB 可用内存）
-   **IT-JSON-005**：JSON Schema 关键字段存在性验证
-   **IT-INT-001**：用户配置生命周期测试（未配置→已配置的字段变化）


### 👤 AI 系统编程专家评审意见

**核心观点**：测试用例对了我上次审查提出的所有关键问题，但有一个严重的遗漏：**错误路径的 JSON 输出格式完全没有测试**。

**对照审查结论的检查清单**：

| 审查问题 | 对应测试 | 状态 |
|----------|----------|------|
| C1: PressureLevel 序列化一致性 | IT-CLI-003 (`--metric pressure`) | ✅ 已覆盖 |
| C2: macOS sysctl C 字符串安全 | 手动场景 2 | ⚠️ 仅靠手动，无自动化崩溃测试 |
| C3: macOS 可用内存虚高 | 手动场景 2 | ⚠️ 仅靠手动 |
| S1: 零内存守卫 | UT-EE-004 | ✅ 已覆盖 |
| S2: MemAvailable fallback | 未覆盖 | ❌ 但 v0.1 暂不实现，可接受 |
| S3: stderr/stdout 分离 | IT-CLI-004 | ⚠️ 仅验证错误情况，未验证正常输出的纯净性 |
| S4: 连续压力建议 | 手动场景 3 | ⚠️ 仅手动 |

**遗漏的关键风险**：错误路径的 JSON 输出。当 `hawk-eye-mem --json` 因权限不足或平台不支持而失败时，它应该输出什么？如果输出的是非 JSON 错误信息，Agent 解析 JSON 的代码会直接崩溃。**失败模式也必须是合法的 JSON**。

**增补要求**：
-   **IT-ERR-001**：权限错误时输出合法错误 JSON
-   **IT-ERR-002**：不支持的平台输出合法错误 JSON


### 👤 Hermes/Agent 生态专家评审意见

**核心观点**：测试用例缺少“Agent 消费视角”的稳定性验证。Agent 不是人类，它依赖字段的精确语义。

**关键问题**：

1.  **`--metric` 输出必须绝对纯净**：IT-CLI-002 验证了 stdout 包含数字，但**没有验证 stdout 只有数字**（不含换行以外的字符）。Agent 通过 MCP 调用时，返回值是字符串，如果带有多余空格或换行，Agent 需要额外处理。这对人类无所谓，对 Agent 是隐患。

2.  **`--interval` JSON Lines 的格式一致性**：IT-CLI-009 验证了输出 2 行 JSON，但**没有验证每行都是独立合法的 JSON**（而非 JSON 数组）。Agent 消费 JSON Lines 时逐行解析，如果某行不合法，Agent 逻辑会中断。

3.  **`confidence` 枚举值稳定性**：Agent 可能写死 `if confidence == "conservative"`。测试应该验证序列化后的字符串值不会意外改变。

**增补要求**：
-   **IT-CLI-002B**：`--metric` 输出无污染验证（stdout 精确匹配数字+换行）
-   **IT-CLI-009B**：JSON Lines 每行独立合法 JSON 验证


### 👤 法务评审意见

**核心观点**：测试用例覆盖了首次运行引导和免责声明，但验证粒度不够。

**具体问题**：手动场景 4 验收标准写了“首先输出免责声明，然后输出快速引导”。但**没有规定免责声明的最小内容**，也没有验证声明中包含“No warranty”关键词。如果实现时免责声明被简化为“Use at your own risk”一句话，法务认为不够。

**增补要求**：
-   **IT-FIRST-001**：首次运行免责声明内容验证（必须包含 “No warranty” 和 “Use at your own risk”）
-   **IT-FIRST-002**：免责声明先于引导输出的顺序验证
-   **IT-FIRST-003**：`.onboarded` 文件创建后不再显示的验证

-   **中**：体验优化或边缘场景完善。


### ✅ 评审结论

**高级产品经理最终裁定**：
测试用例文档 V1.0 结构合理、覆盖基本充分，但存在一个致命盲区：**错误路径的 JSON 输出格式未被验证**。这是 Agent 集成场景下不可接受的。

**准予进入执行，条件如下**：
1.  **必须增补** IT-ERR-001、IT-ERR-002（致命级）。
2.  **强烈建议增补** UT-EE-005、IT-JSON-005、IT-CLI-002B、IT-FIRST-001（高优先级）。
3.  **建议增补** IT-INT-001、IT-CLI-009B（中优先级）。
4.  增补用例需在第 1-2 周编码阶段同步实现，最晚第 3 周集成测试前完成。
