# 秋毫mem · 风险补遗与应对预案 v1.0

**致创始人**：
这份文件的目的，是找出那些"一旦发生，项目就可能停滞或死亡"的风险。我们对每一个风险都给出了缓解措施和明确的决策人。你读完这份预案后，如果仍然决定继续，那么秋毫mem才真正具备了立项的底气。

---

## 👤 产品风险 —— 核心是"信任崩塌"与"价值幻觉"

### 风险1：`safe_max_tokens` 的估算误导，导致产品信用破产

- **问题细化**：我们默认系数针对 Llama 3 8B，但用户可能用 Qwen-72B、DeepSeek-V2 甚至非 Transformer 架构的模型。KV Cache 的计算方式、量化等级 (GGUF/AWQ)、张量并行等，都会让实际单 Token 内存消耗产生数倍偏差。
- **最坏后果**：Agent 根据"安全"建议将上下文调整为 8K，但推理时仍然 OOM，导致正在运行的关键任务（如数小时的代码生成或数据分析）中断且无法恢复。用户会在社区说"秋毫mem 就是个骗子"。
- **缓解措施**：
  - **v0.1 版核心信息降级**：将输出字段 `safe_max_tokens` **降级**为 `estimated_safe_context_window`，并强制在 JSON 和 CLI 输出中高亮 `confidence: "low"`，直到用户主动配置了模型参数。
  - **安全冗余加倍**：在没有用户自定义配置的情况下，安全边际从 10% 提升到 30%。
  - **"逃生舱"模式**：在检测到内存进入 `critical` 压力时，`action` 不再是 `reduce_context`，而是 `stop_and_dump`，建议 Agent 立即保存状态并停止推理。这是最后一道防线。
- **决策人**：产品经理，需在原型阶段就加入"模型配置"的交互设计。

## 👤 技术风险 —— 核心是"跨平台陷阱"与"GPU 黑盒"

### 风险2：macOS 统一内存模型下的数据失真

- **问题细化**：`vm_stat` 显示的 `Pages free` 在 macOS 下极低，因为系统会激进地用作文件缓存。真正的"可用内存"取决于 `memory_pressure` 指标。如果我们简单照搬 Linux 的算法，秋毫mem 在 Mac 上会**永远报内存不足**，失去用户信任。
- **最坏后果**：完全无法在 Apple Silicon（核心早期用户群）上使用。
- **缓解措施**：
  - 将 macOS 的原型开发作为**第二阶段的核心里程碑**，而非可选项。
  - 在 `MemoryCollector` trait 的 macOS 实现中，必须读取并计算 `memory_pressure` 指标，并将其映射到与 Linux 语义对齐的统一 `available_mb` 和 `pressure` 字段。
- **决策人**：技术负责人，需深入研究 Apple 的内存管理文档。

### 风险3：GPU 显存监控的工程复杂度被低估

- **问题细化**：`nvidia-smi` 输出解析脆弱，依赖闭源驱动。AMD 的 `rocm-smi` 生态更不成熟。Apple Silicon 的 GPU 内存使用情况没有官方命令行工具可以简单获取（需要 Metal API）。
- **最坏后果**：核心卖点"感知物理内存"在异构计算场景下，实际只能感知 CPU 内存，价值大打折扣，被社区吐槽为"半成品"。
- **缓解措施**：
  - 在第二阶段，**明确声明 v1.0 仅支持 CPU 内存监控**。
  - GPU VRAM 监控作为独立的 Plugin/Collector 架构在后需版本 (v1.1+) 中支持，并以实验性功能发布。
- **决策人**：技术负责人，需在架构设计时为 GPU Collector 预留接口。

## 👤 市场与运营风险 —— 核心是"为他人做嫁衣"与"昙花一现"

### 风险4：被框架级功能"吞噬"

- **问题细化**：`llama.cpp`、Ollama、vLLM 等推理框架，本身就知道模型加载了多大，占用了多少内存。如果他们在下个版本就增加一个 `--json-status` 输出，秋毫mem 作为外部工具的额外价值在哪里？
- **最坏后果**：框架内置了类似功能后，秋毫mem 瞬间过时，Star 增长停滞，社区流失。
- **缓解措施**：
  - **价值上移**：我们不做一个被动的内存读取器，而是做一个主动的**跨模型、跨框架的资源协调者**。
  - **核心策略**：强化 `agent_guidance` 字段。框架能告诉你"我用了多少"，但只有秋毫mem 能根据整个系统的剩余资源，告诉 Agent **"你还能安全地做什么"**。这是我们的护城河。
  - **生态绑定**：尽快成为 Agent 框架（Hermes）的"官方推荐技能"，从外部工具变为生态组件。
- **决策人**：市场与运营总监，需将传播重点从"监控"转向"决策"。

### 风险5：开源社区的"试用-抛弃"循环

- **问题细化**：项目在 Hacker News 或 Reddit 上火了几天，获得大量 Star，但用户试用一次后就将其遗忘，没有形成日常使用习惯。Agent 的调用频率远低于预期。
- **最坏后果**：高开低走，成为又一个"看起来很酷但没用"的玩具，无力支撑后续商业化。
- **缓解措施**：
  - **锚定高频场景**：在推广中，重点展示在**长时间运行的 Agent 循环** 中的价值，而不是"单次查看内存"。突出 `--interval` 的持续监控能力。
  - **打造"部署清单"心智**：制作高质量的集成指南，让用户觉得"搭建 Agent 开发环境，秋毫mem 是默认安装的一环"。
- **决策人**：市场与运营总监，需策划一个"持续集成"的最佳实践系列内容。

## 👤 法务风险 —— 核心是"全球合规"与"商标盲区"

### 风险6：商标在中国可用，但在国际市场存在暗礁

- **问题细化**："秋毫"拼音 "Qiuhao" 对非中文用户极不友好，难以发音和记忆。英文代号 "Qiuhao Mem" 可能导致我们**被迫在海外市场使用完全不同的品牌**，造成品牌割裂。
- **最坏后果**：在国际化推广时（如 Product Hunt 出海），因为品牌名无法拼读而受阻。更严重的是，可能出现一个海外竞品抢注了英文名，导致我们产品出海时被迫改名。
- **缓解措施**：
  - **立即注册防御性英文商标**：立刻将 "**HawkEye Mem**" 或你认可的其他英文名作为海外商标提交申请。"秋毫" = 明察秋毫 = Hawk-Eye，含义连贯。
  - **主攻中文开发者社区**：在第一年，明确主战场是以中文为主的大模型开发者社区（如掘金、知乎、B站），将国际化优先级后置。
- **决策人**：法务 + 创始人，需立即决策海外品牌策略。

### 风险7："建议"功能导致用户业务损失的赔偿风险

- **问题细化**：虽然我们加了 `confidence` 字段和免责声明，但在某些司法管辖区（特别是重视消费者保护的欧洲），单纯的免责声明可能不足以完全免除因软件建议导致间接损失的责任。
- **最坏后果**：企业用户声称其依赖秋毫mem 的建议调整配置，导致了数万美元的计算资源浪费或业务中断，并提起诉讼。
- **缓解措施**：
  - **使用强免责协议**：在 Apache-2.0 协议之外，增加一个显著的 **AS-IS / 无担保** 条款文件 `DISCLAIMER.md`，并在 CLI 首次运行时要求用户确认。
  - **限制建议范围**：在文档中明确，本工具**仅建议，不自动执行**任何系统级变更。
- **决策人**：法务，需起草一份针对性的免责声明文件。
