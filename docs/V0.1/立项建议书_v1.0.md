## 二、核心定位

**秋毫mem** 是一个 **AI-Native 的内存监控 CLI 工具**，专为本地部署的大语言模型与 AI Agent（如 Hermes、Claude Code、AutoGPT 等）设计。

我们不做更好的 `free` 或 `htop`，而是定位为 **Agent 环境感知层** 的关键组件——输出结构化数据与可决策的语义化建议，使 Agent 能够基于实时物理内存状态，自主调整上下文长度、触发卸载策略或发出预警。

**核心差异化**：
- 输出面向 Agent 决策，而非人类肉眼监控；
- 提供 `safe_max_tokens` 估算与 `action` 语义指令；
- 原生支持 MCP 协议，可作为 Tool 直接注册进 Agent 工作流。

## 三、目标用户

1. 在 MacBook / Linux 工作站上本地部署大模型的开发者

3. 企业推理服务器运维人员（需要轻量、无侵入的内存感知与自动预警）

## 四、第一阶段：调研与立项（已完成）

### 4.1 市场与竞品调研

我们将竞品重新划分为 **数据源工具** 与 **决策层工具**。秋毫mem 属于后者，其真正竞争者并非 `free`、`htop`，而是 LLM 推理框架内建的资源检查能力，以及未来可能出现的"Agent 环境感知"方案。

#### 竞品分析表

| 工具 | JSON 原生输出 | 语义建议 | Agent 调用友好度 | 跨平台 | 性能开销 | 是否集成进 LLM 工作流 |
|------|--------------|----------|------------------|--------|----------|----------------------|
| `free` | ❌ | ❌ | 低 | Linux only | 极低 | 否 |
| `htop` | ❌ | ❌ | 低 | Linux/macOS | 中 | 否 |
| `glances` | 有限 | ❌ | 中（API 过重） | 多平台 | 中 | 否 |
| `nvidia-smi` | 部分 | ❌ | 低 | 仅 NVIDIA | 低 | 否 |
| vLLM 监控 | 有 | 仅限自身 | 中 | 仅服务端 | 低 | 绑定框架 |
| **秋毫mem** | ✅ 原生 JSON | ✅ `safe_max_tokens`, `action` | **极高** | Linux / macOS | <2 MB 常驻 | MCP / CLI 直接调用 |

**核心洞察**：市场存在明确空白——没有任何轻量级工具专门为 AI Agent 提供"剩余内存 → 可采取行动"的语义映射。这既是需求缝隙，也是我们的品类定义机会。

#### 用户痛点清单（10 条，源自社区反馈与调研）

1. Agent 执行大模型推理时因物理内存耗尽 OOM，任务直接崩溃，无法优雅降级。  
2. 开发者必须手动查看 `htop` 决定是否加载更大模型，流程无法自动化。  
3. Agent 框架无法感知实时剩余内存，只能硬编码一个保守的上下文窗口上限。  
4. Apple Silicon 统一内存场景下，剩余容量与可分配给 GPU 的内存关系不透明。  
5. 现有工具输出面向人类，Agent 无法直接从终端获取结构化信息。  
6. 长时间运行的 Agent 循环中，内存泄漏或碎片导致推理速度下降，缺乏预警。  
7. 运维侧希望轻量级、无外部依赖、可被脚本调用的内存检查工具，而非重型监控栈。  
8. 多模型并行加载时，能否安全加载的决策完全靠试错，缺少预判。  
9. 现有 JSON 方案（如 `glances` API）启动重、响应慢，不适合 Agent 高频短周期调用。  
10. 缺少"可回收内存量"估算，Agent 不知道清理缓存能否腾出足够空间。

**痛点真实度判定**：**极高**，特别是 OOM 导致的任务中断带来显著的时间与经济损失，具备强烈的付费改善动机。

### 4.2 技术可行性验证

我们使用 Rust 编写了 MVP 原型，并划定了第一阶段的最小可行范围。

**MVP 范围**：
- **优先支持 Linux**（`/proc/meminfo`），macOS 紧随其后（处理 `vm_stat` 与统一内存模型）。
- 基础指标输出：`total_mb`、`available_mb`、`used_percent`。
- 提供 `safe_max_tokens` 的保守估算公式。
- 稳定 JSON 输出，零外部依赖，交付单个二进制文件。

**跨平台架构**：  
定义了 `MemoryCollector` trait，Linux 和 macOS 分别实现。未来扩展 Windows 或 GPU VRAM 采集仅需增加新实现，上层决策逻辑完全不变。

**safe_max_tokens 估算（v0.1 保守公式）**：
```
safe_max_tokens = floor(available_mb * 1024 * 1024 / (model_kv_cache_bytes_per_token + safety_margin))
```
- 默认 `model_kv_cache_bytes_per_token` = 2048（针对 7B～8B 级别模型的粗略拟合）
- 安全边际 = 10% 的当前可用内存
- 后续版本支持通过配置文件注入模型参数字典，并可依据多次实测自动校准。

**性能基准测试（环境：i7-13700H, Linux 6.5）**：
- 常驻内存（RSS） < 2 MB，虚拟内存 < 10 MB
- 单次采集耗时：p50 = 0.3 ms，p99 = 0.8 ms
- 以 1 秒间隔持续运行 1 小时，CPU 占用 < 0.1%，系统无性能抖动

**结论**：技术完全可行，性能开销极低，架构具备明确的扩展弹性。

### 4.3 与 AI Agent 集成设计

我们围绕 Agent 决策链路，定义了结构化输出模型与集成方式。

#### 数据接口规范 v0.1

```json
{
  "timestamp": "2026-05-18T10:32:15Z",
  "system": {
    "total_mb": 32768,
    "available_mb": 5800,
    "used_percent": 82.3,
    "reclaimable_mb": 1200,
    "pressure": "medium"
  },
  "agent_guidance": {
    "safe_max_tokens": 4096,
    "confidence": "medium",
    "action": "reduce_context",
    "reason": "Available memory below 8GB. Current load may cause OOM with large context."
  }
}
```

- `reclaimable_mb`：估算可回收的缓存/缓冲区内存，引导 Agent 进行清理。
- `pressure`：`low / medium / high / critical`，综合内存压力与历史趋势。
- `confidence`：`high / medium / low`，反映估算可信度，同时作为风险免责的提示。
- `action` 与 `reason`：直接嵌入 Agent 决策链路的语义指令。

#### Hermes Skill 示例（异常流处理）

Skill 使 Agent 在感知到内存压力时 **主动干预**，而非等待用户指令：

> "我注意到当前系统可用内存已降至 5.8 GB，当前上下文长度面临 OOM 风险。建议将上下文令牌数降至 4096，或重启模型释放缓存。是否需要我立即执行调整？"

该逻辑在 `action == "reduce_context"` 时自动触发，实现 Agent 的自我约束与环境适应。

#### MCP 协议适配

秋毫mem 以子进程方式运行，启动极快，可直接注册为 MCP Tool：
```json
{
  "name": "check_memory",
  "description": "Get current system memory status with AI-agent actionable guidance.",
  "inputSchema": { ... },
  "outputSchema": { ... }
}
```
通信层使用 stdio，无需额外守护进程，满足 Agent 高频调用的低延迟要求。
