# 秋毫mem · 技术方案设计书 V1.1 增补意见

**增补编号**：TD-SUP-001
**增补日期**：2026-05-19
**增补事由**：创始人要求评估 DeepSeek-Reasonix 对 DeepSeek-TUI 的可替代性
**增补结论**：不替代，采用混合双轨策略

---

## 一、评估结论

**DeepSeek-Reasonix 不可替代 DeepSeek-TUI。** 两者核心能力维度不同：TUI 强在 Rust 代码生成质量，Reasonix 强在缓存命中与成本控制。Reasonix 基于 TypeScript 生态，对 Rust 的 Cargo 依赖解析、宏展开、trait 系统等高级特性的支持未经验证，存在质量风险。

## 二、混合双轨策略

| 任务类型 | 编码工具 | 占比 |
|----------|----------|------|
| CLI 脚手架、`clap` 参数、`serde` 结构体、配置文件解析、单元测试、文档 | **Reasonix** | ~40% |
| `MemoryCollector` trait、macOS `sysctl` 封装、`GuidanceGenerator` 状态机、估算引擎 | **DeepSeek-TUI** | ~60% |

**成本影响**：API 费用约 ¥50-80/月，比纯 TUI 方案节省约 40%。

## 三、对原方案的具体改动

**原方案（主编码 Agent 部分）**：主编码 Agent 仅列 DeepSeek-TUI。
**改为**：主编码 Agent 采用 **DeepSeek-TUI（质量核心）+ Reasonix（成本优化辅）** 混合模式。

其余框架选型、风险识别、工期预估、测试方案均不变。
