## 一、项目背景

### 1.1 现状分析

秋毫mem 已完成 V0.1-V0.4 四个版本，覆盖了核心能力（内存监控、GPU/温度/Agent 进程检测、环境指纹、远程采集、趋势分析、并发度建议）。但存在一个根本问题：

**秋毫mem 是"被动的"**——它只告诉 Agent"现在内存怎么样了"，但 Agent 不知道"那我该怎么调"。

与此同时，用户（蕉佬）在 Hermes 实际使用中积累了 **极致缓存策略** 的一套玩法（三大铁律、缓存预热、批量合并），这套玩法目前是：
- **写在 Skill 里**（~/skills/mlops/prompt-cache-strategy/）
- **静态的**（不感知系统资源变化）
- **孤立的**（不暴露秋毫mem 的存在）

### 1.2 机会窗口

| 要素 | 现状 | V0.5 目标 |
|------|------|----------|
| 缓存策略 Skill | 已存在，静态，无秋毫mem | 动态缓存策略，绑定秋毫mem |
| 秋毫mem 感知层 | 知道内存压力，不干预缓存 | 直接参与缓存决策闭环 |
| 用户心智 | 秋毫mem = 监控工具 | 秋毫mem = Agent 物理大脑 |
| 社区传播 | 0 次社区推荐 | 自传播生态启动 |

### 1.3 核心洞察

用户不在意"你的工具做了什么"，用户在意"你的工具帮我省了多少钱/避免了什么灾难"。

V0.5 的本质不是"加功能"，是**把秋毫mem 从后台推到用户面前**。通过缓存策略 Skill 这个"鱼饵"，让用户先尝到省钱的甜头，再自然发现秋毫mem 的完整价值。

          │ 内存压力 low  │    │ 缓存命中率   │    │ 固定激进策略  │
          │ → 激进缓存    │    │ 降至 ~60%    │    │ 命中率 ~85%   │

### 3.3 REQ-014：秋毫mem 缓存策略输出

**目标**：秋毫mem 新增一个输出字段/API，告诉缓存策略该用什么模式。

**技术方案**：

```json
// 在 get_memory_status 响应中新增
{
  "system": { ... },
  "agent_guidance": {
    "action": "ok",
    "cache_strategy": {
      "mode": "aggressive",            // aggressive / standard / conservative / emergency
      "reason": "内存充裕（可用 12.8GB/16GB），推荐激进缓存",
      "estimated_hit_rate": 97.5,      // 预估命中率 %
      "max_cache_size_mb": 2048,       // 推荐的缓存上限
      "suggested_clear_threshold_mb": 512  // 可用内存低于此值时自动清缓存
    }
  }
}
```

**决策逻辑**：

| 条件 | 缓存模式 | 命中率 | 说明 |
|------|---------|--------|------|
| 可用内存 > 40% 且 压力 low | `aggressive` | 95-99% | 放心用 |
| 可用内存 20-40% 或 压力 medium | `standard` | 85-90% | 正常用 |
| 可用内存 10-20% 或 压力 high | `conservative` | 70-80% | 省着用 |
| 可用内存 < 10% 或 压力 critical | `emergency` | 0-50% | 保命优先 |

## 四、技术方案概要

### 4.1 系统架构

```
┌─────────────────────────────────────────────────────┐
│                    Hermes Agent                       │
│  ┌─────────────────────────────┐                      │
│  │ hermes-cache-strategy Skill │──── 每次推理后 ────→│
│  │  (动态缓存决策树)           │←── 秋毫mem 状态 ─── │
│  └─────────────────────────────┘                      │
└─────────────────────────────────────────────────────┘
                    │ MCP
                    ▼
┌─────────────────────────────────────────────────────┐
│                  秋毫mem (HawkEye Mem)                │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐           │
│  │ 资源采集  │  │ 缓存策略  │  │ 并发建议  │           │
│  │ (现有)    │  │ (新增)   │  │ (REQ-001)│           │
│  └──────────┘  └──────────┘  └──────────┘           │
└─────────────────────────────────────────────────────┘
```

### 4.2 Skill 与秋毫mem 的交互时序

```
Sequence:
  1. Agent 收到用户请求
  2. Skill 加载，调用 秋毫mem MCP 获取系统状态
  3. 秋毫mem 返回内存压力 + 缓存策略建议
  4. Skill 根据建议设置缓存模式
  5. 推理完成，Skill 计算缓存命中率 + 成本报告
  6. Skill 输出成本报告（含秋毫mem 暴露点）
  → 无秋毫mem：Skill 降级到默认策略，提示安装
```

### 4.3 降级策略

| 场景 | 秋毫mem 状态 | Skill 行为 | 缓存命中率 |
|------|-------------|-----------|-----------|
| 正常 | 运行中，内存充裕 | 激进缓存 | 95-99% |
| 降级1 | 运行中，内存紧张 | 保守缓存 | 70-80% |
| 降级2 | 运行中，危机 | 紧急清缓存 | 0-50% |
| 降级3 | **未安装秋毫mem** | 固定激进策略 | ~60%（无资源感知） |
