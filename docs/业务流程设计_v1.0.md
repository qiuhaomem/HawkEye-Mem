## 一、顶层数据流：一次典型的 Agent 调用

这个流程展示了秋毫mem如何在Agent决策链路中发挥作用。

```mermaid
graph TD
    subgraph Agent Loop
        A[用户: "用8K上下文分析这份代码"] --> B[Agent 接收任务]
        B --> C{在推理前调用秋毫mem}
        C --> D[秋毫mem CLI 启动]
        D --> E[采集系统内存指标]
        E --> F[调用估算引擎计算指标]
        F --> G[生成结构化JSON输出]
        G --> H[Agent 解析JSON]
        H --> I{判断 agent_guidance.action}
        I -- "OK" --> J[以8K上下文执行推理]
        I -- "reduce_context" --> K[主动告知用户降低上下文]
        K --> L[以建议的4K上下文执行推理]
        I -- "critical" --> M[告知用户无法执行，建议释放资源]
    end
```

## 二、系统内部核心流程

秋毫mem本身的执行链路，分为四个阶段。

```
用户/Agent执行命令: hawk-eye-mem --json
    │
    ▼
[1. 初始化阶段]
    │
    ├── 解析命令行参数 (--json, --interval, --count等)
    ├── 加载配置文件 (如果存在，获取模型参数)
    └── 选择并初始化 MemoryCollector (Linux/macOS)
    │
    ▼
[2. 数据采集阶段]
    │
    ├── 调用 platform-specific 系统API
    │   ├── Linux: 读取 /proc/meminfo
    │   └── macOS: 执行 vm_stat + 解析 memory_pressure
    ├── 获取原始指标: MemTotal, MemAvailable, Cached, SwapTotal 等
    └── 组装为平台无关的内部数据结构 SystemMetrics
    │
    ▼
[3. 指标计算与建议生成阶段]
    │
    ├── 计算基础指标: total_mb, available_mb, used_percent
    ├── 估算可回收内存: reclaimable_mb (基于 Cached + Buffers 的保守比例)
    ├── 判定内存压力水位: pressure (low/medium/high/critical)
    ├── 调用估算引擎计算 safe context window
    │   ├── 若用户配置了模型参数: 使用精确公式
    │   └── 若无配置: 使用保守默认值，confidence = "low"
    └── 生成 agent_guidance 语义建议
        ├── 根据 pressure 和 available_mb 映射到 action
        └── 生成人类可读的 reason 字段
    │
    ▼
[4. 输出格式化阶段]
    │
    ├── 封装为统一 JSON 结构 (参见接口规范 v0.1)
    ├── 若指定 --json: 直接输出到 stdout
    ├── 若为人类可读: 格式化彩色终端输出
    └── 退出，返回码 0
```

## 三、核心模块职责分解

| 模块 | 职责 | 关键技术点 |
|---|---|---|
| **CLI入口** | 参数解析、流程编排 | clap crate (Rust) |
| **MemoryCollector Trait** | 定义平台无关的内存采集接口 | trait 抽象，统一 SystemMetrics 结构体 |
| **LinuxCollector** | 解析 `/proc/meminfo` | 文件读取，正则/字符串解析 |
| **MacOSCollector** | 解析 `vm_stat` 和内存压力 | 进程调用，Mach API 封装 |
| **EstimationEngine** | 计算 `estimated_safe_context_window` | 可插拔的公式，支持用户配置注入 |
| **GuidanceGenerator** | 生成 `agent_guidance` 建议 | 状态机映射：pressure + thresholds → action |
| **OutputFormatter** | JSON / 人类可读输出 | serde_json |

## 四、关键决策逻辑：压力分级与 Action 映射

这是秋毫mem的决策核心。`pressure` 由 `used_percent` 和 `available_mb` 共同判定，确保在大内存机器上不会误报，在小内存机器上不会漏报。

> **注意**：以下阈值已根据 v1.1 需求评审整改项调整。`critical` 阈值从 `< 1GB` / `> 95%` 前移至 `< 2GB` / `> 92%`，为 Agent 执行保存操作留出内存余量。action 名 `stop_and_dump` 已更名为 `abort_safely`。

| 压力水位 (pressure) | 判定条件 (满足任一) | 对应 action | Agent 行为建议 |
|---|---|---|---|
| `low` | used_percent < 50% 且 available > 8GB | `ok` | 可安全使用请求的上下文长度 |
| `medium` | used_percent 50-80% 且 available 4-8GB | `monitor` 或 `ok` | 当前安全，建议保持监控 |
| `high` | used_percent 80-92% 或 available 2-4GB | `reduce_context` | 主动建议降级上下文长度 |
| `critical` | used_percent > 92% 或 available < 2GB | `abort_safely` | 立即保存状态，终止推理 |

## 五、边界条件与异常处理

| 场景 | 处理方式 |
|---|---|
| **权限不足**（如无法读取 `/proc/meminfo`） | 输出错误JSON，包含 `error: "permission_denied"`，返回码 1 |
| **不支持的平台** | 编译期或运行时报错，明确提示"仅支持 Linux 和 macOS" |
| **配置文件格式错误** | 忽略自定义配置，回退到默认保守模式，并在 stderr 输出警告 |
| **单次采集耗时异常**（>100ms） | 在 JSON 中附加 `collection_duration_ms` 字段，供 Agent 判断是否延迟过高 |
| **--interval 连续监控时内存压力突变** | 每次采集独立判定，输出完整JSON行，无状态依赖 |

## 七、后续可扩展点 (不做，但架构预留)

- **GPU Collector**：实现 `MemoryCollector` trait 的新实现，采集 VRAM，合并入 `system` 输出。
- **远程采集模式**：当 `--remote` 参数被指定时，`MemoryCollector` 通过 gRPC/HTTP 获取远端机器的指标。
- **插件化建议引擎**：`GuidanceGenerator` 可加载外部 WASM 插件，允许用户自定义决策逻辑。
