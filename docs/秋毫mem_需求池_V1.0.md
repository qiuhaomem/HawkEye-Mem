## REQ-001：Agent 自主并发度调节 —— "物理AI"第一步

> 🔥 **来源**：V0.4 W1-W2 开发过程中发现
> **发现人**：创始人（蕉佬）

---

### 一、需求描述

**痛点**：开发者用AI Agent写代码/跑任务时，经常会遇到一个问题——到底该开几个子Agent并发？开少了效率低，开多了机器直接OOM崩了。

**当前做法**：全靠开发者拍脑袋。"分3路并行吧"——但凭什么是3？如果机器只有2C4G，3路可能直接打死。如果机器是32C128G，3路又太浪费。

**秋毫mem 的核心能力已经在手**：
- 能检测 `cpu_cores`（你机器有几个核）
- 能检测 `total_memory_mb`（你有多少内存）
- 能检测 `available_mb`（你现在还剩多少内存）
- 能检测 `memory_pressure`（你现在压力大不大）

**需求**：秋毫mem 应该提供 **`--suggest-concurrency`** 或类似接口，告诉Agent/开发者：

> "你这台机器是 4C8GB，当前空闲 5.2GB，建议最多开 **3 个子Agent并行**，每个子Agent预留 1.5GB 内存安全空间。"

**一句话**：**让Agent自己知道能开几个分身，不会把自己搞崩。**

---

### 二、价值分析

| 维度 | 价值 |
|------|------|
| 🎯 **精准度** | 基于实时物理资源数据，比开发者"凭感觉定并发数"靠谱100倍 |
| 💰 **成本** | 避免OOM导致的开发中断/任务重跑/机器重启 |
| 🚀 **效率** | 最大化利用资源——"能把机器吃满，又不撑死" |
| 🧠 **"物理AI"** | Agent从"纯逻辑体"进化为"感知物理约束的智能体"——这是秋毫mem的终极叙事 |

---

### 三、可能实现方向

```
hawk-eye-mem --suggest-concurrency [--task-memory 512]
```

输出：
```json
{
  "system": {
    "cpu_cores": 4,
    "total_memory_mb": 8192,
    "available_memory_mb": 5248,
    "memory_pressure": "low"
  },
  "suggestion": {
    "max_concurrency": 3,
    "recommended_concurrency": 3,
    "per_agent_safe_memory_mb": 1500,
    "reasoning": "4 cores available, 5.2GB free. With 1.5GB safety margin per agent, up to 3 parallel agents."
  }
}
```

**关键算法**：
1. 基准：`min(cpu_cores * 0.75, available_mb / task_memory)` 
2. 压力感知：如果 `memory_pressure` 是 high/critical，自动降级
3. 任务类型参数：`--task-memory` 告诉秋毫mem每个子任务预期消耗多少内存
4. 安全余量：保留 20% 内存给系统和突发

---

### 四、与现有能力的结合

| 秋毫mem 能力 | 与并发度建议的关系 |
|-------------|------------------|
| `cpu_cores` | 物理上限——并发数不能超过核心数×2 |
| `available_mb` | 内存上限——每个子Agent需要一定内存 |
| `memory_pressure` | 动态调节——压力大时降并发 |
| **环境指纹** | 跨环境迁移后自动重新计算建议并发数 |
| **远程采集** | 集群场景下，秋毫mem帮调度器决定在哪台机器上开多少个Agent |
| **趋势分析** | 预测内存变化趋势，提前调整并发策略 |

## REQ-002：缓存差距分析与优化建议 —— "向99%命中率冲刺"

> 🔥 **来源**：V0.5 验收后复盘发现（当前实测 95-97%，目标 99%）
> **发现人**：创始人（蕉佬）

---

### 一、需求描述

**痛点**：秋毫mem V0.5 声称 aggressive 模式可达 99%+ 缓存命中率，但实际对比 DeepSeek 平台数据，日命中率仅约 97.3%，自动采集数据（state.db）更只有 95.65%。3-4 个百分点的差距意味着每天多花 $1-2 在未命中缓存上。

**当前做法**：全靠人工翻 DeepSeek 平台看命中率，再自己猜是哪里miss的。

**秋毫mem 已有的能力**：
- `--cache-stats`：能看到今日命中率
- `--record --tokens-processed`：能记录每次推理的 token 消耗
- state.db 有 `input_tokens`/`cache_read_tokens` 按会话粒度

**需求**：秋毫mem 应该提供 **`--analyze-cache-gaps`** 或类似接口，告诉 Agent/开发者：

> "今天的缓存命中率是 97.3%（目标 99%）。主要缺口在：①每轮新对话的头 500 token 无法命中（占 miss 的 40%）②tool 输出变化打断前缀连续性（占 35%）③上下文压缩后新摘要不匹配（占 25%）。建议：增加 SOUL.md 稳定前缀长度至 2K token，降低 temperature 至 0.0。"

**一句话**：**告诉你为什么没到 99%，以及怎么修。**

---

### 二、价值分析

| 维度 | 价值 |
|------|------|
| 🎯 **精准度** | 从"看平台猜原因"到"命令行告诉你缺口分布" |
| 💰 **省钱** | 每提升 1% 命中率，日省 $0.5-1（参考 199M/day 用量） |
| 🧠 **认知闭环** | Agent 能自我诊断缓存效率，形成"采集→分析→优化"循环 |
| 🔧 **可操作** | 不仅告诉你问题，还给出具体修复命令 |

---

### 三、可能实现方向

```
hawk-eye-mem --analyze-cache-gaps --days 7

输出示例：
═══════════════════════════════════════
  缓存差距分析报告 · 过去7天
═══════════════════════════════════════
  实际命中率:    97.3%
  目标命中率:    99.0%
  差距:         1.7% → 每天约 miss 3,400K tokens
  
  🔴 高优先级缺口 (60%)：
    · 新会话前缀冷启动 — miss 占 35%
      原因: 每轮新会话前 300-800 token 从未匹配过
      建议: 固定 SOUL.md 前 2048 token 不变化
    
    · tool 输出波动 — miss 占 25%
      原因: 终端输出变化打断 context 连续性
      建议: 减少 tool 开关频次，没变化的 tool 不要移除
    
  🟡 中优先级缺口 (25%)：
    · 上下文压缩重建 — miss 占 15%
    原因: compress 后摘要文本跟之前不同
    建议: compress 时保留前 1024 token 原样
    
  🟢 低优先级缺口 (15%)：
    · 其他（模型切换、API 重试等）
  
  修复建议:
  1. hermes config set model.temperature 0.0
  2. 优化 SOUL.md 结构（固定前缀 >2048 token）
  3. 减少 tools 动态加载/卸载频次
═══════════════════════════════════════
```

### 四、实现思路

1. **数据源**：读取 `cache_stats.jsonl` + 可选读取 `state.db`
2. **分析维度**：
   - 按时间段（小时/天/周）
   - 按会话类型（weixin/cron/api_server）
   - 按模型（deepseek-v4-flash 等）
   - 按上下文压缩前后对比
3. **输出**：JSON（机器可读） + 彩色终端报告（人类可读）
4. **优化建议**：基于命中率分布，给出对应配置调整命令
